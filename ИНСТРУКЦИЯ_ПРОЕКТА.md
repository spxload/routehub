## 1. ЧТО ЗА ПРОЕКТ

RouteHub — система автоматической маршрутизации трафика в VPN-
клиенте Loon (iOS). Назначение: держать каждый ИИ-сервис
(ChatGPT, Claude, Gemini, Grok, Perplexity) на стабильном узле
для защиты аккаунтов от банов за смену IP; направлять обычный
трафик через быстрейшие узлы; экономить платный обходной трафик;
переживать блокировки РКН; обеспечивать работу звонков и TikTok.

Пользователь — Диана. Подписка Lastdep (~58 узлов). Два профиля
(Диана + муж/жена), только iOS.

ГЛАВНЫЙ ДОКУМЕНТ: `ПЛАН_РЕАЛИЗАЦИИ_v3.md` — в нём всё (этапы,
факты, каталог списков, сверка с документацией, журнал решений).
При любом вопросе по проекту — сверяться с ним.

## 4. КАК ДВИГАТЬСЯ ПО ПЛАНУ — ЭТАПЫ

План состоит из 10 этапов. ВАЖНО: один этап = один чат
(не вести все в одном — чат раздувается). Каждый новый чат
начинать словами: «Работаем по плану RouteHub, Этап X».

Порядок этапов (из плана v3):

**ЭТАП A — Проверки на устройстве.** ПЕРВЫЙ. 14 проверок
(A.1–A.14): инвентаризация подписки, лимиты фильтров, тест
ssid, race condition, QUIC, звонки, TikTok, проверка списков,
метод переключения группы и др. Цель — подтвердить факты ДО
написания файлов. Claude расписывает каждую проверку пошагово
(что нажимать в Loon), Диана выполняет и присылает результаты.
Без Этапа A не начинать B.

**ЭТАП B — Серверная часть.** GitHub-репозиторий,
routehub-server.py (тестировщик + рейтинг), routehub-config.json,
check.yml. Результат — рабочий routehub-ratings.json.

**ЭТАП C — Главный конфиг.** routehub.conf начисто: секции
General, Remote Proxy, Remote Filter, Proxy Group, Rule,
Remote Rule, Host, Script, Plugin, MitM.

**ЭТАП D — Скрипты.** routehub-core.js, routehub-health.js,
routehub-speedtest.js, routehub-netwatch.js, routehub-viewer.js.
Все по стандарту надёжности (Раздел 17 плана).

**ЭТАП H — Спидтест.** routehub-speedtest.js подробно: ночной
Wi-Fi + дневной сотовый, два файла скорости, метки.

**ЭТАП E — Разделение Wi-Fi/сотовая.** ssid-trigger, логика
в скриптах.

**ЭТАП F — Режим РКН.** Детектор маяков, обходные узлы.

**ЭТАП I — Звонки.** Группа RH-Звонки, PROTOCOL,STUN,
логические правила.

**ЭТАП J — TikTok.** Плагин на базе TikTok-Unlock, DNS-маппинг.

**ЭТАП G — Сборка и тест.** Финал: 15 шагов тестирования,
критерии готовности.

После каждого этапа — обновлять план v3 (отмечать сделанное)
и складывать новые файлы в проект.

### Источники списков (проверены, открывались лично)

- Semporia/TikTok-Unlock — разблокировка TikTok.
Полный каталог с оценками — `КАТАЛОГ_СПИСКОВ.md`.

### Ключевые подтверждённые факты Loon (полностью — в плане v3,
Раздел 2 и Приложение 2)
- `setSelectPolicy` / `$config.getConfig(name,select)` —
  переключение группы; точный метод проверить на A.12.
- `$httpClient.get({node:"имя"})` — запрос через узел.
- `$config.getConfig()` отдаёт ssid и running_model.
- `$utils.geoip` — гео по IP в скрипте.
- `ssid-trigger` — разный режим по сети.
- Группы: select, url-test, fallback, load-balance (PCC).
- Логические правила AND/OR/NOT (3.1.7+).
- PROTOCOL: HTTP/HTTPS/TCP/QUIC/STUN/UDP.
- Плагины с `[Argument]` — визуальные настройки в UI.
- `[Host]` — DNS-маппинг по доменам.
- УСТАРЕЛО: `switch-node-after-failure-times`.
- НЕ существует: `timeout`/`max-failed-times` у групп,
  `header=` в [Remote Proxy].
- `network-changed` — работает только ОДИН скрипт.
- Узлы из подписки неизменяемы.

### Правило для списков
Любой список перед добавлением — 5 шагов проверки (открыть
содержимое, понять что это, в какую группу, нет ли лишнего,
честный тег). Детали — в каталоге и плане Раздел 16.
