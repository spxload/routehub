# Этап F — заметки (whitelist РКН + генеральная модель маршрутизации)

> Статус: ГЛАВНЫЙ ВОПРОС ЗАКРЫТ полевым тестом под whitelist 2026-06-12.
> Накопитель решений Этапа F, чтобы будущий чат не терял контекст.

## ✅ WHOOSH — ДИАГНОЗ ПО СКРИНШОТАМ Loon (2026-06-12, под whitelist)
- `api.whoosh.bike:443` ловился `DOMAIN-SUFFIX,whoosh.bike,DIRECT` (старый конфиг,
  до C-draft-25) → статус «Не удалось», TCP, вис 18с. Под whitelist DIRECT мёртв.
  → C-draft-25 переводит whoosh.bike на RH-RU (локальное правило бьёт remote
  whitelist-DIRECT) → под whitelist уйдёт в обход. ЛЕЧЕНИЕ ВНЕСЕНО.
- `app-analytics.whoosh.bike` ловится `DOMAIN-KEYWORD,analytics.,REJECT-DROP`
  (Privacy-список) → DNS-REJECT. НО Диана подтвердила: **Whoosh работает нормально
  даже с блоком рекламы** → этот REJECT приложению НЕ мешает, Privacy трогать НЕ
  нужно. Открытый вопрос №1 (Privacy выше/ниже РФ) — оставляем как есть, REJECT
  аналитики безвреден. Проблема была ТОЛЬКО api.whoosh.bike на DIRECT.

## РЕШЕНИЯ ДЛЯ ДАШБОРД-ЭТАПА (из вопросов Дианы 2026-06-12) — СВЕРЕНО С ДОКОЙ LOON
1. **Добавление доменов через дашборд — ТОЛЬКО через локальный store + личный
   remote-список KV. Конфиг и его [Rule] скрипт НЕ читает и НЕ правит.**
   ДОКА (Script API): `$config.getConfig()` отдаёт группы, выбор политик, ssid,
   режим, FINAL — но НЕ текст правил [Rule]. → дашборд НЕ может «прочитать, что
   Диана добавила правило в интерфейсе Loon». Идея «добавляю в Loon-UI, дашборд
   читает и переносит» — НЕВОЗМОЖНА (нет API чтения [Rule]).
   РАБОЧАЯ СХЕМА (идея Дианы — принята): дашборд кнопкой добавляет домен →
   `$persistentStore.write` локально (pending) → если Worker доступен, POST
   /addrule → пометить выгруженным; если нет (whitelist) — лежит в pending,
   досылается при следующем опросе. Локальный store = источник правды, KV = зеркало.
   ГИБРИД: КРИТИЧНОЕ (нужное и под whitelist, как whoosh) — в ЛОКАЛЬНЫХ правилах
   конфига (Worker под whitelist недоступен); удобное пополнение — личный список KV.
2. **«Не удалось» поймать пассивно НЕЛЬЗЯ — подтверждено докой (Script 6.1).**
   - http-request срабатывает при ЗАПРОСЕ ($response=undefined).
   - http-response срабатывает при ОТВЕТЕ. Но при «Не удалось» (соединение не
     установилось, 0 байт — как api.whoosh.bike на DIRECT) ОТВЕТА НЕТ →
     http-response НЕ вызывается → неудачу скрипт не получает.
   - Оба типа работают только для HTTP/HTTPS и только для ЯВНО ПЕРЕХВАТЫВАЕМЫХ
     правилом доменов, не «весь трафик».
   - Журнал Requests Loon наружу/в store НЕ пишется, API чтения НЕТ.
   ВЫВОД: «слушать неудачи всего трафика» — невозможно. ЕДИНСТВЕННОЕ рабочее —
   АКТИВНЫЙ СТОРОЖ: скрипт/дашборд сам periodically `$httpClient.get({url, node})`
   по ЗАРАНЕЕ ЗАДАННОМУ списку важных доменов; callback `errormsg` (null=успех,
   иначе причина) честно ловит неудачу. Результат → store → дашборд показывает
   «домен: ошибка». Тратит немного трафика, проверяет только список, НЕ перехват.
3. **Авто-триггеры дашборда по опросу Worker — РАБОТАЮТ.** Открытый дашборд каждые
   N сек fetch Worker /dashboard, сравнивает состояние; увидел изменение (подписка
   обновилась / сменился режим) → сам `fetch http://routehub.io/refresh` → Loon
   перехватывает (http-request) → скрипт обновляет. БЕЗ участия Дианы. Ограничения:
   только пока вкладка открыта (нет фонового процесса iOS); только лёгкое (≤10с).

## ✅ РЕЗУЛЬТАТЫ ПОЛЕВОГО ТЕСТА ПОД WHITELIST (2026-06-12, k1, C-draft-23-test)

**ГЛАВНЫЙ ВОПРОС ЗАКРЫТ: Loon ПРОБИВАЕТ член `DIRECT` в `fallback`.**
Тест: группа `RH-Тест = fallback, DIRECT, RH-Обход, url=зарубежный generate_204`,
на неё направлены example.com + tbank.ru. Под реальным whitelist оба ОТКРЫЛИСЬ,
группа ушла с DIRECT на RH-Обход. → Loon тестирует DIRECT url'ом и при его
недоступности переключается. **Модель «слабый DIRECT по умолчанию» жизнеспособна,
без флипов netwatch.** Запасной план (netwatch-флип) НЕ НУЖЕН.

**Прочие подтверждённые факты теста:**
1. **РФ-whitelist домены (gosuslugi, sber, ozon, vk, mail, vtb, alfabank...) —
   на DIRECT под whitelist РАБОТАЮТ.** Переключение их в обход ВРЕДНО: вылезает
   «выключите ВПН», режутся функции (анти-VPN сервиса). → им нужен ЖЁСТКИЙ DIRECT,
   в обход пускать НЕЛЬЗЯ. Открытый вопрос №2 решён в пользу жёсткого DIRECT.
2. **РФ-сервисы НЕ в whitelist (tbank, whoosh, мелкие банки) — на DIRECT под
   whitelist мертвы**, т.к. РКН их не разрешает. tbank ПОЛНОСТЬЮ открылся через
   обход (немецкий узел!), без «выключите ВПН» → геоблока по IP нет, мешал только
   РКН. → отдельная ступень «обход только 🇷🇺» НЕ нужна, общий RH-Обход годится.
   whoosh подтверждён скриншотами (api.whoosh.bike «Не удалось» на DIRECT).
3. **Обход несёт всё иностранное** (ИИ, instagram, bbc, FINAL) — RH-АВТО доходит
   до обходного хвоста корректно. ИИ под whitelist работает через обход (подтв.).
4. **DIRECT по умолчанию (RH-Главный=DIRECT) под whitelist убивает непиннованный
   иностранный трафик** (wikipedia/example умерли) — подтверждает необходимость
   авто-ухода в обход через fallback (что и пробито в п. выше).

## 🎯 ФИНАЛЬНАЯ МОДЕЛЬ (внедрена в C-draft-25)
- **РКН-whitelist домены** (forg-lib: whitelist-domains, hxehex[+Сбер], whitelist-ips)
  → **жёсткий `DIRECT`** (НЕ в обход — анти-VPN). ВЫШЕ всего РФ.
- **Мелкие банки (category-ru) + Apple-список + банки ЦБ** → **RH-RU** (часть не в
  whitelist РКН, на DIRECT под whitelist мертвы; крупные банки пойманы whitelist
  выше → жёсткий DIRECT).
- **GEOIP-RU + РФ-не-whitelist** → **`RH-RU = fallback, DIRECT, RH-Обход`** (норма
  DIRECT; whitelist → обход).
- **FINAL** → **`RH-Главный = fallback, DIRECT, RH-АВТО`**.
- **Личные исключения** (локальные DOMAIN-SUFFIX ВЫШЕ всего, бьют remote whitelist):
  `whoosh.bike → RH-RU`. Растущий список.
- **Явные пины:** ИИ → RH-AI; youtube/instagram/telegram/refilter → RH-АВТО;
  звонки (STUN) → RH-Звонки; реклама → REJECT-DROP.
- **Apple push (17.0.0.0/8)** → RH-RU.
- fallback: **`max-timeout=2000, interval=120`** (вкл. AI-группы — worker v1.6.0).
  `url` пробы — ЗАРУБЕЖНЫЙ generate_204.

## ОТКРЫТЫЕ РЕШЕНИЯ ИЗ C-draft-18 — СТАТУС
1. Privacy выше/ниже РФ-DIRECT — ОСТАВЛЕНО как есть. Whoosh работает с блоком
   рекламы → REJECT аналитики безвреден, менять не нужно.
2. РФ-whitelist жёстко DIRECT vs RH-Прямой — РЕШЕНО: **жёсткий DIRECT** (флип вреден).

## Компромиссы DIRECT-by-default (принять осознанно)
- Нормальный режим: заблокированный, но не попавший в списки сайт идёт DIRECT →
  не откроется. Нужен полный refilter + изредка ручные добавления (личные исключения).
- `url` пробы обязательно ЗАРУБЕЖНЫЙ; лаг переключения = interval (120с).
- Доменные правила раньше IP (Loon 3.9); локальные > подписка — личные исключения
  бьют remote whitelist-DIRECT.
