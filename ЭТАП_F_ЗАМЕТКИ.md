# Этап F — заметки (whitelist РКН + генеральная модель маршрутизации)

> Статус: ЗАПЛАНИРОВАНО. Делать после полевого подтверждения Этапа E.
> Накопитель решений Этапа F, чтобы будущий чат не терял контекст.

## ГЕНЕРАЛЬНАЯ МОДЕЛЬ (кандидат): «слабый DIRECT по умолчанию»
Идея Дианы (2026-06-08), из полевых наблюдений: RH-Apple=DIRECT под
whitelist → ничего не шло; банки на DIRECT под whitelist тоже не все
работают. Вместо «проксировать по умолчанию (FINAL→RH-АВТО), DIRECT
для RU/Apple» — перевернуть: **DIRECT по умолчанию через
самодетектящийся fallback; проксировать/обходить только явно указанное.**

Примитив:
```
RH-Главный = fallback, DIRECT, RH-АВТО, url=http://cp.cloudflare.com/generate_204, interval=300
FINAL,RH-Главный
```
- Нормальный режим: проба DIRECT к зарубежному generate_204 проходит →
  трафик идёт DIRECT (быстро, локально, меньше нагрузки на VPN, меньше
  экспозиции под бан аккаунтов).
- Whitelist: проба DIRECT падает → весь не-пиннованный трафик уходит в
  RH-АВТО → bypass. Единообразно, без костылей на каждую группу.

Явные пины (идут в нормальном режиме мимо DIRECT):
- ИИ-домены → RH-AI.
- Известно-заблокированное (refilter, youtube, instagram, telegram) →
  RH-АВТО (проксируется и в нормальном режиме).
- Звонки (STUN/UDP) → RH-Звонки.
- Реклама → REJECT-DROP.

Снимаются хард-DIRECT пины — поглощаются RH-Главным:
- группа RH-Apple (была select DIRECT/RH-АВТО);
- правило `GEOIP,RU,DIRECT` и запланированный RH-RU;
- (Apple push 17.0.0.0/8 → RH-Главный или просто на FINAL).
Решает разом Apple/RU/банки под whitelist одним механизмом.

## КРИТИЧНО — одна проверка решает ВСЁ
Вся модель висит на: **тестирует ли Loon член `DIRECT` в fallback url'ом
или считает его «всегда доступным»** (док. Loon 4.3 это не уточняет).
- Пробит → модель работает (нормальный DIRECT, whitelist → bypass).
- «Всегда доступен» → RH-Главный всегда берёт DIRECT, под whitelist не
  переключается → модель НЕ работает.
Проверка на k1: под реальным whitelist, либо временно задать `url` на
адрес, заблокированный для DIRECT но открытый через обход, и смотреть,
уходит ли RH-Главный с DIRECT на RH-АВТО/обход.

## Запасной план (если DIRECT в fallback не пробится)
`RH-Главный = select, DIRECT, RH-АВТО`; netwatch по детекту whitelist
флипает на RH-АВТО (нормальный → DIRECT). Менее изящно (зависит от
детекта netwatch), но работает. Аналогично для RU/Apple.

## Компромиссы DIRECT-by-default (принять осознанно)
- Нормальный режим: заблокированный, но не попавший в списки сайт идёт
  DIRECT → не откроется. Нужен полный `refilter` + изредка ручные
  добавления редких доменов. Сейчас (proxy-by-default) такие сайты
  проксируются и работают — это регрессия на «длинном хвосте». Взамен:
  меньше трафика через VPN, быстрее, меньше бан-экспозиция.
- Банки под whitelist НЕ гарантированы в принципе: DIRECT рубит
  whitelist; bypass с ЗАРУБЕЖНЫМ выходом рубит геоблок банка. Шанс
  только у bypass с РУ-выходом (такие узлы в пуле обхода есть, 🇷🇺), но
  fallback не гарантирует попадание именно на них. При необходимости —
  отдельная группа банков на bypass-РУ.
- `url` пробы обязательно ЗАРУБЕЖНЫЙ (иначе под whitelist проба к
  RU-allowlist пройдёт и не переключит); лаг переключения = `interval`.
- Доменные правила раньше IP (Loon 3.9) — явные пины срабатывают до
  любого fallback-дефолта.

## ОТДЕЛЬНАЯ НАХОДКА — FaceTime / QUIC (нормальный режим, чинится сейчас)
Наблюдение Дианы: `disable-udp-ports=443` в [General] рубит UDP/443 и
ломает FaceTime. Звонки критичны.
Решение: **убрать `disable-udp-ports=443`.** Компромисс с YouTube (ради
которого ставился — сбить YouTube с QUIC на TCP для проксирования)
решать отдельно: либо точечное правило `AND,((PROTOCOL,QUIC),
(NOT,((DOMAIN-KEYWORD,googlevideo)))),REJECT-DROP` (план 9b.4), либо
принять QUIC. `disable-stun` НЕ трогать (остаётся false).

## RU-случай (частный экземпляр генеральной модели)
Если генеральную модель не принимаем, точечно по RU:
`GEOIP,RU,RH-RU`, `RH-RU = fallback, DIRECT, RH-Filter-Обход, url=зарубежный generate_204`.
Та же открытая проверка (пробит ли DIRECT). Запасной — select + флип netwatch.
Банки/доменные DIRECT идут раньше (домен > IP).

## Прочее по Этапу F (план, Раздел 9)
