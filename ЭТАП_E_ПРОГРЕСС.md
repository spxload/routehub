## АРХИТЕКТУРА AI: два режима через Worker (C-draft-13, worker v0.7.0+)

Один источник `routehub.conf` (select + скрипты); fallback-вариант собирает Worker
ТРАНСФОРМАЦИЕЙ.
- Базовый линк `?key=kN` — СКРИПТОВЫЙ (RH-AI select + RH-Core/Health/Net).
- `?key=kN&ai=fallback` ИЛИ флаг `ai_fallback` — FALLBACK: RH-AI select->fallback
  (+ RH-Filter-Обход последним), RH-Core/Health/Net off. Приоритет: ?ai= > флаг > скрипт.
- Уведомления о смене узла — НАТИВНЫЕ Loon. `routehub-ai-watch.js` не подключён.
- Пострановые непересекающиеся фильтры (DE/NL/CH/FR/GB/Прочее, NameRegex по флагу +
  lookahead) — группировка в RH-AI; набор стран провизорный (уточнить по счётчикам).

**ВАЖНО:** базовый линк — СКРИПТОВЫЙ. Для теста fallback — линк с `&ai=fallback`.

## ПЕР-СЕТЕВЫЕ УЗЛЫ ЧЕРЕЗ WORKER — РЕАЛИЗОВАНО (worker v0.8.0, netwatch v0.3.0)

Цель (Диана): на Wi-Fi показывать метки ТОЛЬКО Wi-Fi и сортировать по Wi-Fi; на
сотовой — только сотовой. Решение — отдавать узлы через Worker, не через гист.

Почему не гист и не два списка: имена узлов скрипт переименовать не может →
метки печёт сервер. Гист-raw кэшируется CDN (минуты) → рефреш «ложится» с задержкой.
Два предсобранных списка не экономят (Worker всё равно читает гист и выбирает).

Механика (опционально, по умолчанию НЕ трогает рабочую подписку):
- **Worker GET `/nodes?key=kN`** — рендерит узлы под СОХРАНЁННУЮ сеть устройства
  (`devices.json[key].net`): метка только этой сети (🛜 или 📱) + сортировка по её
  скорости; `Cache-Control: no-store` (без CDN-задержки). Узлы без данных этой
  сети — в конце без метки.
- **Worker POST `/net` {key,nonce,net}** — netwatch сообщает текущую сеть. Пишется
  только привязанному устройству при совпадении `nonce` (привязку не создаёт).
- **netwatch v0.3.0** — при смене сети шлёт `/net` (nonce из `rh_nonce`, его создаёт
  спидтест; нет nonce — пропуск).
- **`/config?...&src=worker`** — подписка = Worker `/nodes` + `update-interval`
  (по умолч. 120с; можно `&refresh=N`). Без `src=worker` — гист-raw как раньше.

ЗАВИСИМОСТЬ: переключение метки/сортировки по сети требует, чтобы netwatch (RH-Net)
был ВКЛЮЧЁН = СКРИПТОВЫЙ режим. В fallback-режиме RH-Net выключен → `/net` не шлётся,
`/nodes` рендерит по последней/умолчанию (wifi). Тестировать `src=worker` на
скриптовом линке (без `&ai=fallback`).

Покрытие сотовой: метку на сотовой получают только узлы, измеренные на сотовой
(BATCH_CELL=5 по умолч.). Для меток на всех — флаг `cell_unlim` (cellall).

Тест-линк: `…/config?key=k1&src=worker` (можно `&refresh=60`).

## ОСТАЛОСЬ ПРОВЕРИТЬ В ПОЛЕ
- `src=worker`: Loon принимает Worker-подписку, `/nodes` отдаёт base64, метки одной сети.
- Смена сети: netwatch шлёт `/net`, в течение `update-interval` `/nodes` переключает
  набор; активные соединения не рвутся при переимпорте.
- `update-interval` минимум в Loon 3.3.9 (в доке нет) — подтвердить.
- `$utils.ipaso(ip)` на сотовой; `yandex.ru/internet/` отдаёт IP (`ECHO_URL`).
- Счётчики пострановых фильтров RH-AI.

## ЗАФИКСИРОВАННЫЕ ФАКТЫ (детект сети/оператора, доки 6.2)
1. Контекст между чатами — в файлах.
2. Детект сети — БЕЗ геолокации (`ssid` + маяки; whitelist — `ya.ru` жив).
3. `detectOperator`: основной — Worker `/whoami` (DIRECT, `cf.asOrganization`);
   запасной — `ECHO_URL` → IP → `$utils.ipaso(ip)`.
4. Worker (`*.workers.dev`) при whitelist НЕДОСТУПЕН → whitelist детектит ТОЛЬКО
   netwatch на устройстве (маяки). Это и решило: netwatch — авторитет по сети.
5. У скрипта НЕТ API перекачать подписку (только интервал/`loon://update`/ручное).
6. `setSelectPolicy` — рабочий (select); fallback-группой скрипт НЕ управляет.

## РЕШЕНИЕ ПО ОПЕРАТОРУ В WHITELIST
Локально `$utils.ipaso(ip)`. Схему «IP → Worker через узел» не строим.
