## СОСТОЯНИЕ ЭТАПА E

### Уже сделано (в рамках Этапа D)
- **Детект типа сети** — `routehub-netwatch.js` v0.2.0 (тип `network-changed`,
  единственный — И4). Слои: `ssid` из `$config.getConfig()` → маяки
  `generate_204` (gstatic + Cloudflare cp) через DIRECT → `ya.ru` через DIRECT.
- **Изоляция обходных узлов.** Группа `RH-Обход` (`fallback, RH-Filter-Обход`)
  существует, но в `[Rule]` на неё НЕ указывает ни одно правило → обходной
  трафик = 0 на ЛЮБОЙ сети (не только Wi-Fi). Маршрутизация на `RH-Обход`
  при whitelist — задача Этапа F, не E.

### Осталось в E
1. **Решение по `ssid-trigger`.** С учётом решения «любой Wi-Fi = без
   whitelist» (ниже) именованный домашний SSID в `ssid-trigger` НЕ нужен.
   Рекомендация — **В1: не добавлять `ssid-trigger`** (на всех сетях нужен
   режим `RULE`; параметр ничего не добавляет). В2 (всё `RULE`) — только как
   страховка от ручного перевода Loon в глобальный директ/прокси. Финальный
   выбор В1/В2 — за Дианой.
2. (Опционально) `[Host]` DNS по SSID — пока не требуется.

### РЕШЕНИЕ ПО Wi-Fi (Диана)
**Любой Wi-Fi считается сетью без whitelist (доверенная).** Следствия:
- список домашних SSID для целей whitelist НЕ нужен;
- текущее поведение netwatch уже соответствует (на Wi-Fi маяки/whitelist не
  проверяются — `net='wifi'` сразу при наличии `ssid`);
- размен (принят осознанно): на whitelisted-Wi-Fi в РФ (редкий случай)
  система не уйдёт на обход на этой сети. Вероятность мала (домашний Wi-Fi
  обычно не в whitelist), выигрыш — простота.

## ЗАФИКСИРОВАННЫЕ ФАКТЫ (детект сети и оператора)

> Согласовано в обсуждении; относится к E и к F.

1. **Контекст между чатами не теряется существенно.** `СТАРТ.md` +
   `ПЛАН_РЕАЛИЗАЦИИ_v3.md` + репозиторий + память несут состояние.
   Теряется только «живая» возня сессии. Дисциплина файлов = механизм
   передачи контекста.

2. **Детект сети работает БЕЗ геолокации (координат).**
   - Wi-Fi / сотовая — по наличию `ssid` + маяки `generate_204` через DIRECT.
   - whitelist РКН — общие маяки молчат, а `ya.ru` через DIRECT отвечает
     (рунет жив, внешний интернет порезан).
   - Разрешение iOS «Локация: Всегда» нужно ТОЛЬКО чтобы Loon отдал имя
     `ssid`; это не определение страны.

3. **Детект оператора (`detectOperator` в netwatch):**
   - Основной путь: DIRECT-запрос на Worker `/whoami`. Cloudflare видит IP
     телефона (запрос DIRECT) и возвращает `request.cf.asOrganization`.
     Запрос именно DIRECT, НЕ через узел (через узел Worker увидел бы IP узла).
   - Запасной путь: DIRECT на `ECHO_URL` (`yandex.ru/internet/`) → IP из тела
     → оператор ЛОКАЛЬНО через `$utils.ipaso(ip)`.

4. **Worker (`*.workers.dev`) при whitelist РКН НЕДОСТУПЕН.** В обычном
   режиме доступен, `/whoami` работает. (Факт от Дианы.)

5. **`request.cf.asOrganization` нельзя получить локально на телефоне** —
   это обогащение Cloudflare, существует только внутри Worker для IP
   соединения (дока Cloudflare). И его нельзя применить к произвольному
   переданному IP (`cf` относится только к IP соединения).

6. **`request.cf.asOrganization` НЕ «лучше» определяет сеть, чем
   `$utils.ipaso`** — это один механизм (IP → ASN-lookup), разница лишь в
   свежести базы ASN. Для крупных операторов РФ (МТС/Мегафон/Билайн/Tele2/
   Yota) результат практически совпадает.

## РЕШЕНИЕ ПО ОПЕРАТОРУ В WHITELIST-РЕЖИМЕ

Схему «IP от Яндекса → на Worker через узел с `?ip=` → классификация на
сервере» НЕ строим. При whitelist Worker недоступен (факт 4), а определять
оператора проще ЛОКАЛЬНО через `$utils.ipaso(ip)` — этот путь уже в коде.
Гонять IP на Worker через узел ради чуть более свежей базы ASN не оправдано
(сложность + лишний трафик + зависимость от живости узла). Текущий
`detectOperator` уже покрывает оба случая: `/whoami` когда Worker доступен,
`$utils.ipaso` когда отрезан. **Дописывать ничего не нужно.**

## ВОПРОСЫ ДИАНЫ → ВЫВОДЫ

### 1. Пострановые фильтры для RH-AI (визуальная группировка) — для F/полировки
Идея: вместо «глобально по скорости, страны вперемешку» — отдельные
`[Remote Filter]` по стране, перечислить по порядку в `RH-AI`; внутри страны
порядок сохранится скоростным.
- **Реализуемо.** Имя узла начинается с флага-эмодзи (`publish_nodes.flag_of`),
  тег `[VPN]` — внутри имени. Фильтр: `NameRegex` вида `🇩🇪.*\[VPN\]`.
  Узлы внутри фильтра сохранят порядок `nodes-kN` (Worker-сортировка по
  скорости) → «страна за страной, внутри по скорости».
- **НЕ подтверждено документацией** (раздел 2.6/4.3): что фильтры в группе
  идут строго в перечисленном порядке, а внутри фильтра сохраняется порядок
  подписки. На практике обычно так — ПРОВЕРИТЬ на устройстве.
- **Чисто визуально.** `RH-AI` — `select`, узел назначает `routehub-core.js`
  (Германия-якорь, sticky). Пострановые фильтры порядок ВЫБОРА и sticky НЕ
  меняют — только вид списка при ручном просмотре.
- **RH-АВТО не трогаем.** Он на `RH-Filter-Осн` с глобальным скоростным
  порядком (`fallback` = быстрейший живой). Менять глобальную сортировку
  `nodes-kN` НЕЛЬЗЯ (иначе RH-АВТО возьмёт быстрейший-в-первой-стране, а не
  глобально быстрейший). Группировка — ТОЛЬКО в RH-AI отдельными фильтрами.
- **Цена:** по фильтру на страну; новая страна без своего фильтра выпадет из
  группировки. Лёгкий вариант: 3-5 основных стран явными фильтрами +
  «прочие `[VPN]`» последним. Полное дробление противоречит принципу простоты.
- Вывод: полировка отображения, не функция. Делать — лёгким вариантом, по
  желанию; логичнее в F (или отдельным мелким правком конфига).

### 2. RH-Обход: переключается ли скриптом; частота проверки РКН — для F
- **Сейчас RH-Обход скриптом НЕ переключается** и трафика не получает
  (нет правила, нет вызова из скриптов). Маршрутизация на обход при
  whitelist — Этап F, механизм не выбран.
- **Правила `[Rule]` статичны; скрипт их переписать не может** (дока 6.2 —
  скрипт меняет только ВЫБОР в `select`-группе и режим `setRunningModel`).
  → «увести трафик на обход» динамически через правила нельзя.
  Рабочий путь для F — **переключаемая select-группа-шлюз**:
  напр. `RH-Шлюз = select, RH-АВТО, RH-Обход`; на неё указывает `FINAL`
  (и, возможно, AI), скрипт при whitelist ставит выбор `RH-Обход`
  (`$config.setSelectPolicy`), иначе `RH-АВТО`. Детали — в F.
- **Частота проверки РКН — НЕ каждую минуту:**
  - netwatch срабатывает по СОБЫТИЮ смены сети, не периодически; whitelist
    может включиться без смены сети (та же сотовая) → событие пропустит.
  - поминутный опрос — расход батареи/трафика, ненадёжная минутная гранулярность cron.
  - лучше: реагировать на ОТКАЗ (узлы внешки умерли + общие маяки молчат +
    `ya.ru` жив → whitelist) + лёгкая периодическая проверка на сотовой в
    такте health (5 мин). Решаем в F.

## ОТКРЫТЫЙ ВОПРОС НА ГРАНИЦЕ E/F — РЕШЁН
Был: netwatch считает любой Wi-Fi сетью `wifi` и не проверяет whitelist на
Wi-Fi. РЕШЕНО решением Дианы «любой Wi-Fi = без whitelist» — текущее
поведение netwatch намеренно и корректно. Уточнение «незнакомый Wi-Fi →
проверять как сотовую» НЕ вводим.
