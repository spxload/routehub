## СОСТОЯНИЕ ЭТАПА E

### Уже сделано (в рамках Этапа D)
- **Детект типа сети** — `routehub-netwatch.js` v0.2.0 (тип `network-changed`,
  единственный — И4). Слои: `ssid` из `$config.getConfig()` → маяки
  `generate_204` (gstatic + Cloudflare cp) через DIRECT → `ya.ru` через DIRECT.
- **Изоляция обходных узлов.** Группа `RH-Обход` (`fallback, RH-Filter-Обход`)
  существует, но в `[Rule]` на неё НЕ указывает ни одно правило → обходной
  трафик = 0 на ЛЮБОЙ сети. Маршрутизация на `RH-Обход` при whitelist —
  задача Этапа F.

### Решения по E (закрыто)
1. **`ssid-trigger` — В1: НЕ добавлять (подтверждено Дианой).** На всех сетях
   нужен режим `RULE`; параметр ничего не добавляет.
2. **Любой Wi-Fi = сеть без whitelist (Диана).** Список домашних SSID для
   whitelist не нужен; текущее поведение netwatch уже соответствует.
   Размен (принят осознанно): на whitelisted-Wi-Fi в РФ (редкий случай)
   система не уйдёт на обход на этой сети.
3. `[Host]` DNS по SSID — не требуется.

Сетевое разделение Wi-Fi/сотовая на этом закрыто. Остаток — задел под F.

## ЗАФИКСИРОВАННЫЕ ФАКТЫ (детект сети и оператора)

1. **Контекст между чатами не теряется существенно.** `СТАРТ.md` +
   `ПЛАН_РЕАЛИЗАЦИИ_v3.md` + репозиторий + память несут состояние.
   Дисциплина файлов = механизм передачи контекста.
2. **Детект сети — БЕЗ геолокации.** Wi-Fi/сотовая по `ssid` + маяки
   `generate_204` (DIRECT); whitelist — общие маяки молчат, `ya.ru` (DIRECT)
   отвечает. «Локация: Всегда» нужна ТОЛЬКО чтобы Loon отдал `ssid`.
3. **`detectOperator`:** основной путь — DIRECT на Worker `/whoami`
   (Cloudflare видит IP телефона, отдаёт `request.cf.asOrganization`);
   запасной — DIRECT на `ECHO_URL` (`yandex.ru/internet/`) → IP из тела →
   `$utils.ipaso(ip)` локально.
4. **Worker (`*.workers.dev`) при whitelist РКН НЕДОСТУПЕН.** В обычном
   режиме доступен. (Факт от Дианы.)
5. **`request.cf.asOrganization` нельзя получить локально** — обогащение
   Cloudflare, только внутри Worker для IP соединения; нельзя применить к
   произвольному переданному IP.
6. **`cf.asOrganization` НЕ «лучше», чем `$utils.ipaso`** — один механизм
   (IP→ASN), разница лишь в свежести базы ASN; для крупных операторов РФ
   совпадает.

## РЕШЕНИЕ ПО ОПЕРАТОРУ В WHITELIST-РЕЖИМЕ
Схему «IP от Яндекса → на Worker через узел» НЕ строим. При whitelist Worker
недоступен; оператора проще определять локально `$utils.ipaso(ip)` — путь уже
в коде. Текущий `detectOperator` покрывает оба случая. Дописывать не нужно.

## ВОПРОСЫ ДИАНЫ → ВЫВОДЫ

### 1. Пострановые фильтры для RH-AI (визуальная группировка)
- **Реализуемо, чисто визуально.** Имя узла начинается с флага-эмодзи
  (`publish_nodes.flag_of`), тег `[VPN]` внутри имени. Фильтр: `NameRegex`
  вида `^🇩🇪.*\[VPN\]`. Узлы внутри фильтра сохранят порядок `nodes-kN`
  (Worker-сортировка по скорости) → «страна за страной, внутри по скорости».
- `RH-AI` остаётся `select`, узел назначает `routehub-core.js` (Германия-якорь,
  sticky). Фильтры порядок ВЫБОРА и sticky НЕ меняют — только вид списка.
- **RH-АВТО не трогаем** (остаётся на `RH-Filter-Осн`, глобальный скоростной
  порядок). Глобальную сортировку `nodes-kN` менять НЕЛЬЗЯ.
- **«Правка конфига» = изменения ТОЛЬКО в `routehub.conf`** (добавить строки
  `[Remote Filter]` + переписать строку `RH-AI`); без скрипта и сервера.
- **Лёгкий старт для теста на устройстве:** два фильтра — `RH-Filter-DE`
  (`^🇩🇪.*\[VPN\]`) + существующий `RH-Filter-Осн`;
  `RH-AI = select, RH-Filter-DE, RH-Filter-Осн`.
  ПЕРЕД коммитом в живой конфиг проверить на устройстве:
  (а) фильтры идут в перечисленном порядке; (б) **дедупликация** — Loon
  показывает узел один раз или дублирует между фильтрами (НЕ документировано).
  Если дублирует — нужен catch-all с исключением сгруппированных стран
  (негативный lookahead), сложнее. Поэтому в живой конфиг не пишем до теста.

### 2. RH-Обход: переключение и частота РКН (для F)
- **Сейчас RH-Обход скриптом НЕ переключается** и трафика не получает.
- **Правила `[Rule]` статичны; скрипт их переписать не может** (дока 6.2 —
  только ВЫБОР в `select`-группе и `setRunningModel`). Рабочий путь F —
  переключаемая select-группа-шлюз (`$config.setSelectPolicy`).
- **Частота — НЕ каждую минуту.** netwatch = по событию смены сети; whitelist
  может включиться без смены сети. Лучше: реагировать на ОТКАЗ (узлы внешки
  мертвы + общие маяки молчат + `ya.ru` жив) + лёгкая периодическая проверка
  на сотовой в такте health (5 мин).

### 3. AI при whitelist: почему НЕ fallback (важно, финализировать в F)
- **Проблема реальна:** `RH-AI` собран только из `[VPN]`-узлов; при whitelist
  они мертвы → AI не работает, хотя обходные узлы AI часто тянут.
- **Предложение Дианы:** перевести `RH-AI` в `fallback`, обход последним
  фильтром, скрипт убрать.
- **Вывод: `RH-AI` в `fallback` НЕ переводим.** Причины:
  - `fallback` не умеет sticky: нет `tolerance`/гистерезиса (дока 4.3 —
    «первый доступный», ретест каждый интервал). Любой временный сбой
    `generate_204` на текущем узле → прыжок на другой → смена IP → риск бана
    (приоритет #1 проекта, Этап D).
  - `fallback` тестит `generate_204`, не AI-доступность → может сидеть на
    «живом, но регион-блок» узле. Скрипт использует серверный рейтинг
    (per-AI green) + AIM-скорость.
  - «Германия первой» во `fallback` = «первый живой DE по `generate_204`», а
    не «лучший AI-green DE».
  - При снятии whitelist `fallback` прыгает обратно на DE → ещё смена IP.
  - «`fallback` быстрее скрипта» — да, но для AI быстрая смена узла = вред;
    инерция (sticky) — это фича. Нужную скорость реакции на whitelist даёт
    netwatch (смена сети) + health, без потери sticky.
  - Документально только связка «скрипт над `select`» выражает
    sticky+hysteresis+cooldown+same-country+AI-green; `fallback`/`url-test` —
    нет.
- **Принято (финализировать в F):** оставить `RH-AI = select` + скрипт
  (sticky); ДОБАВИТЬ обходные узлы как кандидаты ПОСЛЕДНЕГО РЕЗЕРВА,
  задействуемые ТОЛЬКО при whitelist; селектор остаётся sticky и в обходе;
  при снятии whitelist — возврат на DE с cooldown (против флапа).
  Зависимость: сервер должен оценивать обходные узлы на AI (план B.4.7) —
  ПОДТВЕРДИТЬ в коде `routehub-server.py` на Этапе F.
