## АРХИТЕКТУРА AI: два режима через Worker (C-draft-13, worker v0.7.0)

Решено (Диана): один источник `routehub.conf` (select + скрипты); fallback-вариант
собирает Worker ТРАНСФОРМАЦИЕЙ — два файла не плодим.
- Базовый линк `?key=kN` — СКРИПТОВЫЙ (RH-AI select + RH-Core/Health/Net).
- `?key=kN&ai=fallback` ИЛИ флаг `ai_fallback` в devices.json — FALLBACK:
  RH-AI select->fallback (+ RH-Filter-Обход последним, url+interval=600),
  RH-Core/Health/Net -> enabled=false. Приоритет: явный ?ai= > флаг > скрипт.
- Уведомления о смене узла — НАТИВНЫЕ Loon. `routehub-ai-watch.js` БОЛЬШЕ НЕ
  ПОДКЛЮЧЁН (файл оставлен в репо, но не используется).
- Пострановые НЕПЕРЕСЕКАЮЩИЕСЯ фильтры (DE/NL/CH/FR/GB/Прочее, NameRegex по
  флагу + lookahead) — группировка в RH-AI; набор стран ПРОВИЗОРНЫЙ (уточнить
  по счётчикам в Loon; пустые безвредны, Прочее ловит остальное).

**ВАЖНО:** базовый линк теперь СКРИПТОВЫЙ. Для теста fallback — линк с `&ai=fallback`.

**Статус теста fallback (AI):** 6 ч — узел не менялся, сессии не рвались
(при стабильном верхнем DE fallback де-факто sticky). Нужно увидеть событие
отказа верхнего узла и whitelist.

## ПЕР-СЕТЕВАЯ СКОРОСТЬ (wifi/cell) И ОБНОВЛЕНИЕ — ОТКРЫТО

- Метрики хранятся раздельно: metrics-kN.json → по узлу {w:{...}, c:{...}}.
  nodes-kN сейчас сортируется по down (приоритет wifi), НЕ по текущей сети.
- **У скрипта НЕТ API перекачать подписку** (дока 6.2). Обновление nodes-kN под
  текущую сеть: интервал времени / Shortcuts / ручной тап `loon://update?sub=all`.
  Тихо скриптом — нельзя.
- **Диана тестирует Shortcuts** (автоматизация на смену сети -> loon://update).
  Если сработает надёжно — следующий шаг: эндпоинт Worker `/net?key=&net=`
  + пересортировка nodes-kN по скорости текущей сети (w/c уже есть).
- Скриптовый режим решает это нативно (netwatch + локальные speed_wifi/cell +
  setSelectPolicy), без перекачки подписки — это его сильная сторона.

## Что скрипт может с fallback-группой (дока 6.2)
Только ЧИТАТЬ: `getSelectedPolicy`, `getSubPolicies`, `getConfig().policy_select`;
+ проба через узел (`$httpClient {node}`), пуши, `setRunningModel`, переключение
ДРУГИХ select-групп. УПРАВЛЯТЬ выбором самой fallback-группы НЕЛЬЗЯ
(переоценивает каждый interval). Поэтому шлюз whitelist (F) — select, не fallback.

## ЗАФИКСИРОВАННЫЕ ФАКТЫ (детект сети и оператора)
1. Контекст между чатами не теряется существенно — файлы несут состояние.
2. Детект сети — БЕЗ геолокации (`ssid` + маяки; whitelist — `ya.ru` жив).
3. `detectOperator`: основной — Worker `/whoami` (DIRECT, `cf.asOrganization`);
   запасной — `ECHO_URL` -> IP -> `$utils.ipaso(ip)`.
4. Worker (`*.workers.dev`) при whitelist НЕДОСТУПЕН.
5. `cf.asOrganization` локально не получить; 6. это тот же механизм, что `$utils.ipaso`.

## РЕШЕНИЕ ПО ОПЕРАТОРУ В WHITELIST
Локально `$utils.ipaso(ip)` (путь уже в коде). Схему «IP -> Worker через узел» не строим.

## ОСТАЛОСЬ ПРОВЕРИТЬ В ПОЛЕ
- `$utils.ipaso(ip)` на сотовой; `yandex.ru/internet/` отдаёт ли IP (`ECHO_URL`).
- `update-interval` для `[Remote Proxy]` в 3.3.9 и его минимум (в доке нет).
- Shortcuts: надёжно ли автоматизация на смену сети запускает `loon://update?sub=all`.

## ПОСТРАНОВЫЕ ФИЛЬТРЫ / RH-Обход / частота РКН (для F)
- Имя узла = имя провайдера (#-фрагмент); флаг-эмодзи в начале, тег [VPN] внутри.
  Фильтр по флагу = фильтр по имени провайдера. Loon дедуплицирует узлы
  между фильтрами в группе (проверено Дианой), но держим явную партицию.
- **RH-Обход скриптом не переключается.** Правила [Rule] статичны. Для F —
  переключаемая select-группа-шлюз. Если fallback-AI приживётся — whitelist для AI
  решается нативно (обход последним фильтром).
- **Частота РКН — НЕ каждую минуту:** реакция на отказ + лёгкая периодика (5 мин).
