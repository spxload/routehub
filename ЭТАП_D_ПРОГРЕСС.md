## ГРАБЛИ (подтверждено — не наступать заново)
1. **Узлы — base64** (publish_nodes.py). Worker atob/btoa.
2. **Loon нормализует пробелы в именах** → матч по
   `norm = s.replace(/\s+/g,' ').trim()`. Иначе tested:0.
3. **`getSubPolicies(group, cb)` отдаёт JSON-СТРОКУ**, не массив.
   Парсить. Элементы `{"type":"node","name":"..."}`; имя в `.name`.
4. **Проба `__down?bytes=0` → "Empty response data"** (пустое тело).
   Мерить на `bytes=1`.
5. **`ssid` из `getConfig()` часто пуст** на iOS → детект сети по ssid
   ненадёжен; пусто → 'cell'. Надёжный детект — D.7 (netwatch).
6. **Loon кэширует remote-скрипты по URL** → Worker добавляет `?v=<ts>`
   к script-path (ВРЕМЕННО, снять в D.9). Задержка доставки ~3–5 мин.
7. **Версия в логе** (`var VERSION`) — обязательна, бамп при правке.
8. **rtt — НЕ пинг.** Это время полного запроса через узел (TCP+TLS+ответ),
   всегда выше пинга, первый «холодный» даёт всплеск. Берём min из 3 проб.
   Для сравнения узлов — годится; как абсолют — нет.

## Поведение скорости/сети (РЕШЕНО этой сессией)
- **Обе метки в имени.** Имя узла несёт ОБЕ: `база · 📶<wifi>↓ 📱<cell>↓`
  (то, что измерено). Снимает проблему «мгновенного переключения» —
  переворачивать список под сеть не нужно, обе цифры всегда видны.
  Телефон шлёт ОБА кэша (wifi+cell), Worker вписывает обе.
- **Сортировка nodes-kN — по Wi-Fi** (домашняя сеть основная). Правильный
  узел под ТЕКУЩУЮ сеть выбирает скрипт-селектор (D.5) из локального кэша
  нужной сети — это и есть «мгновенность», но на уровне ВЫБОРА, не подписи.
- **Имена обновляются только при перекачке подписки** Loon (по интервалу),
  не мгновенно — предел Loon (имена из подписки неизменяемы скриптом).
  Мгновенный просмотр текущих скоростей — задача `routehub-viewer.js`.
- **Переотправка убрана.** POST только после реального замера. Ночной
  cron: первый тик меряет всё → POST; остальные тики — fresh → без POST.
- **rtt в имя НЕ пишем** (раздувает); хранится в кэше для выбора и viewer.

## Покрытие сотовой
- Wi-Fi мерит все узлы за прогон. Сотовая: батч 5 — берёт каждый раз
  СЛЕДУЮЩИЕ 5 «несвежих» (due-логика по 24ч-кэшу) → за ~13 дневных
  сотовых прогонов покрывает все; покрытые держатся 24ч.
- **`cell_unlim:true`** (поле у ключа в devices.json) → сотовая мерит все
  за один прогон. Рекомендуется для безлимитной сотовой (k1) — особенно при
  паттерне «1 переключение на сотовую в день»: событийный замер по 5 не
  покроет, а cell_unlim даёт полный охват за один утренний переход.
- Дневной сотовый замер запускает netwatch (D.7) при переходе на сотовую;
  до D.7 — ручной запуск на сотовой.

## Что МОЖЕТ скрипт при смене сети (network-changed, D.7)
- Узнать сеть (с оговоркой по ssid, грабли №5), писать флаг в store.
- **Перевыбрать узел** select-группы под новую сеть (мгновенно, из кэша).
- Запустить лёгкий сотовый замер.
- Переключить режим Loon (`$config.setRunningModel`) — для детектора РКН.
- Послать `$notification`.
- Делать запросы через узлы (`$httpClient ... node`).
НЕ может: переписать конфиг/фильтры/группы/URL подписки/имена узлов;
сменить тип сети. Только ПЕРВЫЙ network-changed скрипт срабатывает →
один диспетчер. iOS может пропустить событие, если приложение выгружено.

## Нативная реакция Loon на сеть (без скрипта)
- `ssid-trigger` в `[General]`: режим по сети (`"cellular":PROXY,
  "default":RULE`, именованный SSID). Ключи `cellular`/`default` работают
  по ТИПУ сети — SSID не нужен (а он часто пуст, грабли №5).
- `[Host]` с `ssid:` — DNS по сети.
- Меняет только РЕЖИМ и DNS; пересобрать/переранжировать узлы или сменить
  подписку Loon сам не может (это уровень скрипта/групп).

## РКН-аварийность — делать НАТИВНО, не скриптом (Этап F)
- «Live»-переход на обход при whitelist — через нативный `fallback`:
  обычные узлы ПЕРВЫМИ, обходные 🙏 — ПОСЛЕДНИМИ. Loon сам уводит на обход
  за секунды (свой интервал проверки), без ожидания cron.
- Экономия сохраняется: обход включается только когда все обычные мертвы
  (= whitelist). Тюнить `interval`/`tolerance` от дёрганья на кратких сбоях.
- Cron-детектор РКН (маяки) — вторичный «умный» слой; мгновенность даёт Loon.

## Спидтест и списки правил — защита (для D.5/D.9/F)
- Запрос пинится к узлу (`node:`), правила маршрутизации его, как правило,
  не перенаправляют. НО `REJECT`-правило на домен теста (`speed.cloudflare.com`)
  теоретически может уронить → 0 на всех. **ПРОВЕРИТЬ на устройстве** при
  включённых списках, что обычные узлы дают ненулевую скорость.
- Защита: правило-исключение `DOMAIN-SUFFIX,speed.cloudflare.com,...` ВЫШЕ
  reject-списков; в спидтест добавить «**не перезаписывать хорошее значение
  нулём**» (сбой/whitelist → оставить прошлое, не портить кэш).
- При реальном whitelist обычные честно дают 0 (не доходят до Cloudflare) —
  спидтест в этом режиме ПРОПУСКАТЬ (связать с детектором РКН, Этап F).

## Контракты (для D.5+)
- **Аргумент спидтеста:** `"<key>|<origin>|<opts>"` (opts=`cellall` при
  cell_unlim).
- **POST /speed:** `{key, nonce, wifi:[{name,down,rtt}], cell:[{name,down,rtt}]}`.
- **Метрика в имени:** `база · 📶<wifi.down>↓ 📱<cell.down>↓` (что есть).
  Стабильный ключ узла = имя до первого ` · ` (`baseName/stripMetric`).
- **Локальный кэш на телефоне:** `rh_speed_wifi`, `rh_speed_cell` в
  `$persistentStore`: `{ baseName: {down, rtt, ts} }`.
- **Переключение select-группы (A.12):** `$config.getConfig(policyName, selectName)`
  (true/false). Подтвердить на устройстве при первом запуске core.
