## 1. ДИАГНОСТИКА GEMINI — ЗАВЕРШЕНА

**Проблема:** под аккаунтом Дианы Gemini выдаёт «страна не
поддерживается» на ЧАСТИ узлов; другие AI (ChatGPT/Claude/Grok/
Perplexity) на тех же узлах работают.

**Проверены и опровергнуты ВСЕ гипотезы** (2 прогона рейтинга 80
узлов + полные отметки Дианы +/- по двум скриншотам):
- ASN/провайдер — нет (один ASN H2nexus: Польша работает, Финляндия нет).
- Тип IP (datacenter/vpn/residential) — нет (один тип, разный результат).
- Страна — нет (Германия и в «+», и в «-»).
- Транспорт (ws/tcp/grpc), маскировка (sni/fp), IPv6/IPv4 — нет.
- Стабильность узла — нет (узлы со стабильностью 90.8 тоже в «-»).
- **SmartDNS-подмена провайдером — нет.** Финальный прогон с
  DNS-диагностикой: у ВСЕХ 80 узлов dns_check = honest
  (gemini.google.com -> Google AS15169 142.251.x.2; chatgpt.com ->
  Cloudflare AS13335). Провайдер НЕ подменяет DNS ни на одном узле.

**ОКОНЧАТЕЛЬНЫЙ ВЫВОД:** причина — репутация конкретного выходного
IP в глазах Google ПОД АККАУНТОМ. Два узла одного провайдера в одной
стране Google видит по-разному. Анонимный сервер этого определить НЕ
может (страница отдаёт 200 для региона независимо). Готового способа
нет даже у эталонов (lmc999). Аккаунтную доступность Gemini проверяет
только сам пользователь, открыв приложение.

**Объяснение провайдера (подтверждает картину):** провайдер ставит
AI-домены (TikTok/Gemini/Grok/Claude) «в один маршрут» — заворачивает
их на подобранные годные выходы; маскировка трафика «под Cloudflare» —
это обход РКН на ВХОДЕ (чтобы DPI пропустил канал), НЕ меняет выходной
IP, который видит Google. Поэтому на основных узлах Gemini работает, а
на части (обход/игры/экзотика) — нет.

**Корреляция (практически полезная):** основные VPN-узлы (reality и
обычные VPN #1..#N) у Дианы по Gemini работают ВСЕ; «-» только у
обходных, игровых и редких экзотических стран на крупных VPN-ASN
(M247, WorkTitans). RU-узлы под аккаунтом работают (но RU в блок-листе
Gemini ОСТАВЛЕН — на RU работает только под активной сессией, новый
вход с RU-IP Google заблокирует).

## 2. АРХИТЕКТУРА AI-ГРУППЫ (РЕШЕНО)

**Принцип:** не тестировать аккаунтную доступность (невозможно), а
ДЕРЖАТЬ AI на узлах региона, который заведомо работает.

- AI-группа собирается ТОЛЬКО из основных VPN-узлов (НЕ обход, НЕ игры)
  — фильтр NameRegex, напр. ^(?!.*(Обход|Игры)).
- **Германия — основная, всегда первая** (узлов больше всех: 7 + reality,
  все Gemini «+»).
- **+ 2 резервные страны**, выбираются ДИНАМИЧЕСКИ: из стран с >3 узлов,
  сначала по наибольшему числу узлов, при равенстве — по скорости.
  Кандидаты по текущим данным: Польша (4), Финляндия (4); далее
  Нидерланды/Эстония/Турция (по 3). США — только запас (датацентр,
  Google к таким IP придирчивее в долгую).
- Внутри выбранной страны — быстрейший узел (по локальному спидтесту).

**Путь реализации — ГИБКИЙ (выбор Дианы):** select-группа + cron-скрипт-
селектор. Причина: состав подписки меняется (сегодня у одних стран
больше узлов, завтра у других), фиксированный фильтр это не отслеживает.

## 3. СКРИПТ-СЕЛЕКТОР AI (Этап D) — ЧТО ДЕЛАЕТ

Cron-скрипт (периодически). Использует подтверждённый API Loon:
- $config.getSubPolicies(group) — список узлов-кандидатов группы.
- скорости узлов — из $persistentStore (пишет спидтест, Этап H).
- страна узла — из имени или $utils.geoip(out_ip).
- логика: сгруппировать живые узлы по странам -> Германия фикс ->
  + 2 страны (по числу узлов, при равенстве по скорости) ->
  быстрейший узел в наборе -> назначить $config.getConfig(group, node).

**ОГРАНИЧЕНИЕ Loon (важно):** скрипт НЕ может менять СОСТАВ группы и НЕ
может создавать группы — в API только чтение (getSubPolicies) и выбор
(getConfig). Поэтому: в конфиге ОДНА AI-группа с фильтром = пул всех
основных VPN-узлов; скрипт среди пула сам выбирает и назначает финальный
узел. НЕ три группы. Группа — пул, скрипт — мозг выбора.

## 4. ОСТАЛЬНОЙ ТРАФИК (YouTube и пр.) — БЕЗ СКРИПТА

Штатные url-test группы Loon: сами меряют латентность своих узлов и
берут быстрейший живой. Кастомный рейтинг для остального трафика НЕ
нужен. Скрипт-логика только для AI (нестандартный выбор).

---

## 5. СКОРОСТЬ / ПИНГ — ЛОКАЛЬНО НА ТЕЛЕФОНЕ

- Серверный пинг нерепрезентативен (облако != телефон Дианы 50-100 мс).
  Сервер пинг/скорость НЕ меряет.
- Латентность для выбора — штатные url-test группы (дёшево, все узлы).
- Реальная скорость (Мбит/с) — спидтест (Этап H): ночью на Wi-Fi+зарядке
  полный замер, днём по сотовой выборочный. Батчами, прогресс в
  $persistentStore, не успел — домеряет следующей ночью.
- Скорость ЛОКАЛЬНА у каждого устройства (разные сети у Дианы/жены/детей);
  общий рейтинг скорости бессмыслен. Каждый телефон выбирает свой
  быстрейший узел.

## 6. СЕРВЕР — ОСТАВЛЕН (лёгкий фон)

Роль смещена: не «главный мозг», а фон. Делает то, что телефон делает
плохо или дублировал бы 4 раза (свойства узла — общие для всех устройств):
- **живость** узла (бинарно, из облака валидно);
- **стабильность во времени** (EWMA за недели — телефон столько не копит);
- **классификация**: страна / тип IP / ASN / провайдер (для фильтров);
- **новые узлы за 24 ч** (см. п.10 — последнее, что доделать);
- AI-доступность ПОНИЖЕНА в весе: ChatGPT по факту (compliance работает),
  Claude/Grok/Perplexity/Gemini — по стране (грубо). Для AI-группы это
  почти не используется (берём Германию, она работает).

Причина оставить сервер: «сервер думает, телефон исполняет» — тяжёлое
тестирование 80 узлов один раз для всех, не сажая батарею 4 телефонов.

## 7. КОНФИГ — ОДНА ССЫЛКА ДЛЯ ВСЕЙ СЕМЬИ

- Конфиг на GitHub раздаётся ОДНОЙ ссылкой (Loon Remote Config): жена,
  дети подключаются по ссылке — подгружается ВСЁ (узлы, группы, фильтры,
  скрипты, настройки локальных тестов). Чтение публичное, без токена.
- Изменения (фильтры, regex, группы) Диана правит ПРЯМО В GitHub
  (браузер/приложение — она умеет), Loon у всех подтягивает свежую версию.
- Фильтры/группы — в УДАЛЁННОМ конфиге (не в локальном Loon на телефоне),
  иначе изменения не разойдутся семье. Нельзя «менять в Loon на телефоне»
  И «чтобы у всех» одновременно — Loon-фильтр локален.

## 8. ТОКЕН GitHub НА ТЕЛЕФОНЕ — ЗАПРЕЩЁН (категорически)

Конфиг раздаётся по ссылке -> токен записи внутри увидят все, кому дана
ссылка -> любой пишет в репозиторий. Поэтому:
- «выгрузка конфига из Loon на GitHub по кнопке» — НЕТ (нужен токен).
- запись скорости/оценок с телефона прямо в репозиторий — НЕТ.
- Подтверждает вывод 7 проекта. Если когда-нибудь понадобится
  централизованная телеметрия — только через промежуточный приёмник
  (Cloudflare Worker, токен на воркере, не на телефонах). Сейчас НЕ нужно.

## 9. КНОПКИ «НЕ РАБОТАЕТ» — ЛОКАЛЬНО (Этап D)

- Реализация: набор generic-скриптов (каждый = иконка-кнопка, тап вручную).
  Отдельной «панели с кнопками» в Loon нет.
- Тап пишет штраф текущему узлу по сервису в $persistentStore — ЛОКАЛЬНО,
  не на сервер. Выбор узла учитывает штраф.
- Проверка перед занижением: скрипт может проверить узел через
  $httpClient.get({node}) и занизить только при подтверждённой проблеме
  («узел лёг / медленно / не успело переключиться»). НО аккаунтный Gemini
  «страна не поддерживается» анонимно НЕ проверяется — для него проверка
  не подтвердит (доверять нажатию или занижать мягко).
- Проверка должна учитывать режим маршрутизации (обход/белые списки):
  обычный узел и не должен открывать вне-белосписочное — не занижать за
  это ошибочно. ДЕТАЛЬ для Этапа D, когда будет конфиг групп. Не гадать.
