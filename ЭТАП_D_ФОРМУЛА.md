# ЭТАП D — СПЕКА ФОРМУЛЫ ОЦЕНКИ УЗЛОВ

> Источник истины по метрикам/формуле/тирам. Читать из прогресса.
> Принцип: ПРОСТОТА. Пороги привязаны к реальным границам восприятия,
> не «с потолка»; веса стартовые и настраиваемые, валидируются на устройстве.

## 1. Метрики (что меряем через узел на speed.cloudflare.com)
| Метрика | Источник | Влияет на | Статус |
|---|---|---|---|
| down — загрузка, Мбит/с | `__down?bytes=N` | видео, скачивание | есть |
| rtt — отклик без нагрузки, мс | min из 3 проб `bytes=1` | веб, первый токен AI, звонки | есть |
| jit — джиттер, мс | max−min тех же проб | звонки, видеосвязь | v0.4.13 |
| bl — отклик под нагрузкой (bloat), мс | проба ВО ВРЕМЯ скачивания минус rtt | видео без рывков, звонки | v0.4.13 (approx) |
| up — отдача | POST `__up` | исходящий голос/выгрузка | не делаем (мало нужно) |
| потери пакетов | — | звонки | НЕЛЬЗЯ по HTTP |

bl — приблизительно: малая проба пускается параллельно скачиванию; на медленных
(сотовых) каналах проба завершается во время закачки = «под нагрузкой». На очень
быстрых каналах закачка может кончиться раньше пробы → bl может быть null. Валидировать.

## 2. Нормировка метрик в 0–1 — АБСОЛЮТНЫЕ пороги «с запасом» (НЕ относительно соседей)
ПОЧЕМУ не относительно максимума: при 100/25/3 Мбит и делении на лучший (100=100%)
узлы 25 и 3 попали бы в один низкий тир, хотя 25 для видео ХВАТАЕТ, а 3 — нет.
Поэтому достаточность под задачу с «потолком»:
- `s = min(1, down/20)`        # 20 Мбит/с = с запасом на 1080p; 100 и 25 → оба 1.0; 3 → 0.15
- `r = clamp((300−rtt)/(300−30), 0, 1)`   # ≤30мс=1, ≥300мс=0
- `j = clamp((100−jit)/(100−5), 0, 1)`    # ≤5мс=1, ≥100мс=0
- `b = clamp((200−bl)/200, 0, 1)`         # рост задержки под нагрузкой: 0=1.0, ≥200=0
- `стаб` = stability сервера /100 (0–1).
Побочно это честно гасит «палки»: на слабом сигнале у всех узлов низкий s → «видео
тут не пойдёт», а не «лучший из плохих покрашен зелёным».

## 3. Гейт (рейтинг сервера) — фильтр, НЕ слагаемое
Сначала отсев по серверу (двухслойная модель: сервер думает, телефон меряет):
- живой и не red (`light`≠red, не dead);
- для AI — нужная страна (Германия/резерв);
- для звонков — `udp:true`;
- давно не подтверждён живым → не выбирать.
Затем — ранжирование по баллу среди выживших. `стаб` входит малым членом (мигающий
узел опустить), но как множитель/штраф, не как основной вклад.

## 4. Балл по группам (стартовые веса, настраиваемые)
- **AI (RH-AI):**     `0.45·r + 0.30·стаб + 0.15·s + 0.10·j`
- **АВТО (веб/видео/соцсети):** `0.45·s + 0.25·r + 0.20·b + 0.10·стаб`
- **Звонки (RH-Звонки):** `0.40·j + 0.30·r + 0.20·b + 0.10·s`, гейт по UDP.
Профилей ровно три — по числу групп. Отдельные профили под YouTube/Instagram/сайты
НЕ нужны: все стримы хотят одно (пропускная+низкий bloat = АВТО), веб нетребователен
и работает на узле, годном для видео. Делить веб/стрим на разные узлы — только если
позже реально понадобится (группа `RH-Стрим` + правила доменов), сейчас лишнее.

## 5. Тиры (видимая иконка) — абсолютные, по достаточности скорости
4 уровня на 3-сегментной полоске (вариант 2): провайдер использует 🟢⭐🚀💫 — наши
полоски с ними не конфликтуют и при сортировке убывают ровно.
- `▰▰▰` down ≥ 15 (1080p-готов)
- `▰▰▱` 6–15 (720p/SD)
- `▰▱▱` 2–6 (веб/аудио)
- `▱▱▱` < 2 (плохо)
- `⛔` мёртвый
В имени по сети: `🛜▰▰▱ 📱▰▱▱`. Цифры (down/rtt) — за флаг `show_rtt`.
Видимый тир = достаточность СКОРОСТИ (интуитивно «сколько канала»); умный ВЫБОР узла
идёт по баллу группы (п.4), он точнее тира.

## 6. Связь всего (поток)
1. Телефон меряет down/rtt/jit/bl на текущей сети(+операторе) → metrics.
2. Сервер даёт гейт (живость/страна/udp/стаб).
3. Селектор группы: отсев по гейту → балл по профилю → sticky/hysteresis (AI) → выбор.
4. Имя узла показывает тир-полоску (достаточность) по каждой сети.

## 7. Нормировка и условия (важно)
- Баллы считаются по АБСОЛЮТНЫМ порогам → сравнимы между прогонами/местами/операторами
  (не зависят от соседей в замере). Относительный максимум НЕ используется.
- На сотовой значение релевантно текущему оператору → кэш по оператору (D.7), чтобы
  показывать/использовать данные нужной симки.
- EWMA (флаг) сглаживает дрожание во времени; распространён на down/rtt/jit/bl.

## 8. Что где реализуется
- Измерение jit/bl, хранение, тиры-полоски — спидтест v0.4.13 + Worker v0.4.16 (СДЕЛАНО).
- Композит-балл, гейт, профили, sticky — селектор `routehub-core.js` (D.5).
- Кэш/детект по оператору — netwatch (D.7).
- Веса/пороги — валидировать на устройстве, правятся здесь.

## 9. Контракт данных
- POST ITEM (хороший): `{name, down, rtt, jit, bl?}`; мёртвый: `{name, dead:true}`.
- metrics-kN.json: `{matchKey:{w:{down,rtt,jit,bl},c:{...}}}`.
- Кэш телефона: `{baseName:{down,rtt,jit,bl,ts,att,fails}}`.
