# ЭТАП D — СПЕКА ФОРМУЛЫ ОЦЕНКИ УЗЛОВ

> Источник истины по метрикам/формуле/тирам/иконкам. Принцип: ПРОСТОТА; пороги — по
> РЕАЛЬНЫМ битрейтам и порогам Cloudflare AIM, не «с потолка»; веса стартовые,
> валидируются на устройстве. Выводы подтверждены research (см. п.11).

## 0. Две РАЗНЫЕ задачи (не путать)
- **ВЫБОР узла — относительный, БЕЗ отбраковки по скорости.** Всегда ранжируем и
  берём лучший доступный (отсекаем только мёртвых/гейт). Узел не «бракуется» за
  низкую скорость. Относительная «хорошесть в условиях» живёт ЗДЕСЬ.
- **ПОКАЗ (иконка) — два слоя:** (а) абсолютный УРОВЕНЬ качества (что потяну: 360p…4K,
  насыщение на 4K); (б) относительный ПРОЦЕНТ от быстрейшего узла сети — на КАЖДОМ узле
  (для скачивания/торрентов, где скорость не насыщается).

## 1. Метрики (через узел на speed.cloudflare.com)
| Метрика | Источник | Влияет | Статус |
|---|---|---|---|
| down — загрузка, Мбит/с | `__down?bytes=N` (для видео — 2–4 потока, см. п.11) | видео, скачивание | есть (многопоток — TODO) |
| rtt — отклик без нагрузки | min/p50 из ≥3 проб `bytes=1` | веб, первый токен AI, звонки | есть |
| jit — джиттер | разброс проб (max−min) | звонки, видеосвязь | v0.4.13 |
| bl — отклик под нагрузкой (bufferbloat) | проба ВО ВРЕМЯ скачивания − rtt | видео без рывков, звонки | v0.4.13 (approx) |
| потери пакетов | — | звонки | НЕЛЬЗЯ по HTTP (звонковый тир приблизителен) |

## 2. Реальные требования к каналу (битрейты воспроизведения)
360p≈0.7 · 480p≈1 · 720p≈2.5 · 1080p≈5 · 1440p≈10 · 4K≈15 (реком.), 25 комфортно.
Правило: канал ≥ битрейт × 1.5–2 (ABR + буфер). Торренты/скачивание НЕ насыщаются.
ВЫВОД: «видео-готов» ≈5 Мбит (1080p), «4K с запасом» ≈25 Мбит.

## 3. Нормировка метрик 0–1 — абсолютные пороги с НАСЫЩЕНИЕМ на «достаточно с запасом»
- `s = min(1, down / CAP)`, CAP — потолок задачи С ЗАПАСОМ (research: запас обязателен):
  **видео CAP=15** (1080p×запас→4K-вход); **веб CAP=10**; **AI/звонки CAP=3**.
  Запас важен: при просадке ×5 узел 25→5 (видео цело), 5→1 (умерло) — высокий балл у тех, кто переживёт дрейф.
- `r,j,b` — по точечным порогам Cloudflare AIM (п.5), не линейные clamp.
- `стаб = stability/100`.

## 4. Гейт (рейтинг сервера) — фильтр, НЕ слагаемое
Отсев: живой и не red; для AI — нужная страна; для звонков — udp:true; давно не подтверждён → не брать.
Среди выживших — ранжирование по баллу. `стаб` — малый член/штраф. Гейт НЕ бракует живые узлы за низкую скорость.

## 5. Балл по профилям + пороги AIM (research, cloudflare/speedtest)
Очки за метрику (суммируются), пороги Cloudflare AIM:
- **rtt (мс):** <10:+20 · 10–20:+10 · 20–50:+5 · 50–100:0 · 100–500:−10 · ≥500:−20
- **bl=loadedLatencyIncrease (мс):** те же пороги, что rtt
- **jit (мс):** <10:+10 · 10–20:+5 · 20–100:0 · 100–500:−10 · ≥500:−20
- **down (Мбит/с):** <1:0 · 1–10:+5 · 10–50:+10 · 50–100:+20 · ≥100:+30
- потери: нам недоступны → 0 (сценарий не ломается)

Профильные веса (стартовые, настраиваемые):
- **AI:**     `2·rtt + 1.5·jit + 1·bl + 0.5·s(CAP=3)` + проверка доступности ChatGPT/Claude через узел
- **Веб:**    `1.5·rtt + 1·bl + 0.5·jit + 1·s(CAP=10)`
- **Видео:**  `2·s(CAP=15, мерить 2–4 потока) + 0.5·rtt + 0.5·bl`
- **Звонки:** `2·bl + 2·jit + 1·rtt`, гейт UDP (потери недоступны → приблизительно)
Профилей столько, сколько групп; отдельные под YouTube/веб НЕ нужны (узел, годный для видео, годен и вебу).

## 6. ИКОНКА показа — УРОВЕНЬ (заполняющийся блок) + ПРОЦЕНТ (надстрочный), на каждом узле
ОГРАНИЧЕНИЕ: имя узла в Loon — только Unicode-символы (SVG/картинку вставить НЕЛЬЗЯ), цвета нет.
- **Уровень качества — вертикальный блок фикс. высоты, заполнение снизу** (насыщение на 4K=25 Мбит):
  `▁` 360p (<1) · `▃` 480p (1–2) · `▅` 720p (2–5) · `▇` 1080p (5–15) · `█` 4K (15–25) · `█⁺` 4K с запасом (≥25)
- **Процент от быстрейшего узла сети — надстрочными цифрами**, на КАЖДОМ узле: `¹⁰⁰ ⁸² ⁴⁰ ¹¹`
  (= down_узла / max_down_сети · 100; для торрентов/скачивания — главный ориентир)
- **Мёртвый:** `⛔` · **не измерен:** меток нет
- Формат имени: `<имя провайдера> · 🛜<блок>[⁺] <%над> 📱<блок>[⁺] <%над>`
  Пример: `🇩🇪 Германия #4 · 🛜█⁺ ¹⁰⁰ 📱▅ ³⁸`
- Запасные символы (если `⁺` мелко/не отрисуется на iOS): `█✦` или `█↑`. Проверить на устройстве.
- Точные цифры down↓rtt — за флаг `show_rtt` (дополнительно к иконке).
ВАЖНО: блок = абсолют (что потяну), процент = относит. (кто быстрее сейчас). Два слоя в одной иконке.

## 7. Связь всего (поток)
1. Телефон меряет down/rtt/jit/bl на текущей сети(+операторе) → metrics.
2. Сервер даёт гейт (живость/страна/udp/стаб).
3. Селектор: гейт → балл профиля → sticky/hysteresis (AI) → выбор (без отбраковки по скорости).
4. Имя показывает блок-уровень + процент по каждой сети.

## 8. Что где реализуется
- jit/bl, иконки — спидтест v0.4.13 + Worker (тиры-полоски v0.4.17 → ОБНОВИТЬ на блок+процент).
- Композит-балл, гейт, профили, sticky — селектор `routehub-core.js` (D.5).
- Многопоточный замер видео (2–4 потока) — спидтест (D.5/H).
- Кэш/детект по оператору, процент считается в пределах сети+оператора — netwatch (D.7).
- Веса/CAP/пороги — стартовые; валидировать на устройстве, править здесь.

## 9. Контракт данных
- POST ITEM (хороший): `{name, down, rtt, jit, bl?}`; мёртвый: `{name, dead:true}`.
- metrics-kN.json: `{matchKey:{w:{down,rtt,jit,bl},c:{...}}}`.
- Кэш телефона: `{baseName:{down,rtt,jit,bl,ts,att,fails}}` (+ `rh_home_aso`).
- Процент = down / max(down по живым узлам той же сети) · 100, округление до целого.

## 10. Сглаживание и замер (research)
- EWMA α≈0.2 раздельно по Wi-Fi/каждой SIM; хранить разброс; для тира — консервативная
  оценка (скорость = медиана/нижний перцентиль, rtt = p50). Гистерезис ~50 мс против флаппинга.
- Один поток через туннель с RTT занижает скорость в 2–5× (BDP) → для видео-замера 2–4 потока,
  малый объём (5–10 МБ на сотовой, реже). Это исправляет занижение дальних узлов.

## 11. Итоги research (подтверждено)
- Запас (headroom) обязателен: канал ≥ битрейт × 1.5–2; «4K с запасом» ≈25 Мбит. → CAP видео 15, высший тир ≥25.
- Формула AIM (cloudflare/speedtest, `internalConfig.ts`) — переиспользуем пороги (п.5). Сценарии-суммы:
  стриминг `[15,20,40,60]`, звонки `[5,15,25,40]`, игры `[5,15,25,30]`.
- Абсолютная шкала с насыщением ПРАВИЛЬНЕЕ относительной для тира (несвежие замеры; «лучший сам шумит»);
  относительный процент — только как второй слой/тай-брейкер (наш надстрочный %). Подтверждает п.0/п.6.
- Ни один прокси-клиент не строит композит под задачу (sing-box/Surge/Clash — latency+tolerance;
  clash-speedtest — скорость+отсечки). Наш профильный балл — ниша.
- Переиспользовать: cloudflare/speedtest (пороги AIM), xiecang/speedtest-clash (p50/p90 + ChatGPT-unlock),
  hskrasek/cloud-speed (AIM в CLI). Звонки точно не измерим (UDP-потери) — тир приблизителен.
