# ЭТАП D — СПЕКА ФОРМУЛЫ ОЦЕНКИ УЗЛОВ

> Источник истины по метрикам/формуле/тирам. Принцип: ПРОСТОТА; пороги — по
> РЕАЛЬНЫМ битрейтам, не «с потолка»; веса стартовые, валидируются на устройстве.

## 0. Две РАЗНЫЕ задачи (не путать)
- **ВЫБОР узла — относительный, БЕЗ отбраковки по скорости.** Всегда ранжируем и
  берём лучший доступный (отсекаем только мёртвых/гейт). Узел не «бракуется» за
  низкую скорость — для сёрфинга/720p хватает и 3–4 Мбит. Отдельная группа «медленных»
  НЕ нужна. Относительная «хорошесть в условиях» живёт ЗДЕСЬ.
- **ПОКАЗ (тир) — абсолютный, по РЕАЛЬНЫМ порогам.** Отвечает «что потяну»: 1080p /
  720p / SD-веб / плохо. Честно: на слабом сигнале у всех низкий тир — это правда.

## 1. Метрики (через узел на speed.cloudflare.com)
| Метрика | Источник | Влияет | Статус |
|---|---|---|---|
| down — загрузка, Мбит/с | `__down?bytes=N` | видео, скачивание | есть |
| rtt — отклик без нагрузки | min из 3 проб `bytes=1` | веб, первый токен AI, звонки | есть |
| jit — джиттер | max−min проб | звонки, видеосвязь | v0.4.13 |
| bl — отклик под нагрузкой | проба ВО ВРЕМЯ скачивания − rtt | видео без рывков, звонки | v0.4.13 (approx) |
| up / потери | — | — | не делаем / нельзя по HTTP |

## 2. Реальные требования к каналу (битрейты воспроизведения, общеизвестно)
360p≈0.7 · 480p≈1 · 720p≈2.5 · 1080p≈5 · 4K≈15–20 Мбит/с. Веб-сёрфинг — сотни Кбит/с.
ВЫВОД: 4 Мбит/с = уверенные 720p (почти 1080p), НЕ «очень низко». Порог «видео-готов» ≈5.

## 3. Нормировка метрик 0–1 — абсолютные пороги с НАСЫЩЕНИЕМ на реальном «достаточно»
- `s = min(1, down / CAP)`, где CAP — реальный потолок задачи:
  **АВТО/видео CAP=6** (≈1080p; выше — лишнее, решают отклик/джиттер);
  **сёрф/AI CAP=2** (тексту/вебу больше не надо).
  → 100 и 25 → s=1.0 (оба «с запасом»); 6 → 1.0; 3 → 0.5; 1 → ~0.17. Сравнимо в любых условиях.
- `r = clamp((300−rtt)/(300−30), 0, 1)`; `j = clamp((100−jit)/(100−5),0,1)`;
  `b = clamp((200−bl)/200,0,1)`; `стаб = stability/100`.
ПОЧЕМУ насыщение, а не доля от соседа: при 100/25/3 деление на лучший слило бы 25 и 3
в один тир, хотя 25 для видео хватает. Потолок «с запасом» правит это и сравним между местами.

## 4. Гейт (рейтинг сервера) — фильтр, НЕ слагаемое
Отсев: живой и не red; для AI — нужная страна; для звонков — udp:true; давно не подтверждён → не брать.
Затем ранжирование по баллу. `стаб` — малый член/штраф (мигающий опустить), не основной вклад.
ВАЖНО: гейт отсекает мёртвых/неподходящих, но НЕ отбраковывает живые узлы за низкую скорость.

## 5. Балл по группам (стартовые веса, настраиваемые)
- **AI:**     `0.45·r + 0.30·стаб + 0.15·s + 0.10·j`  (s с CAP=2)
- **АВТО:**   `0.45·s + 0.25·r + 0.20·b + 0.10·стаб`   (s с CAP=6)
- **Звонки:** `0.40·j + 0.30·r + 0.20·b + 0.10·s`, гейт UDP.
Профилей три (по числу групп). Отдельные под YouTube/веб НЕ нужны: узел, годный для видео,
годен и вебу; стримы хотят одного (пропускная+низкий bl) = АВТО. Делить — только если позже реально надо.

## 6. Тиры показа — абсолютные, РЕАЛЬНЫЕ битрейты (3-сегментная полоска)
- `▰▰▰` down ≥ 5 (1080p)
- `▰▰▱` 2–4 (720p)
- `▰▱▱` 1 (SD/веб)
- `▱▱▱` < 1 (плохо)
- `⛔` мёртвый
В имени по сети: `🛜▰▰▱ 📱▰▱▱`. Цифры (down↓rtt) — за флаг `show_rtt`.
Видимый тир = достаточность скорости (видео-линза, интуитивно «что потяну»); умный ВЫБОР — по баллу группы (п.5).

## 7. Связь всего (поток)
1. Телефон меряет down/rtt/jit/bl на текущей сети(+операторе) → metrics.
2. Сервер даёт гейт (живость/страна/udp/стаб).
3. Селектор: гейт → балл по профилю → sticky/hysteresis (AI) → выбор (без отбраковки по скорости).
4. Имя показывает абсолютный тир по каждой сети.

## 8. Что где реализуется
- jit/bl, тиры-полоски (реальные пороги) — спидтест v0.4.13 + Worker v0.4.16+ (СДЕЛАНО).
- Композит-балл, гейт, профили, sticky — селектор `routehub-core.js` (D.5).
- Кэш/детект по оператору — netwatch (D.7).
- Веса/CAP/пороги — стартовые; валидировать на устройстве, править здесь.

## 9. Контракт данных
- POST ITEM (хороший): `{name, down, rtt, jit, bl?}`; мёртвый: `{name, dead:true}`.
- metrics-kN.json: `{matchKey:{w:{down,rtt,jit,bl},c:{...}}}`.
- Кэш телефона: `{baseName:{down,rtt,jit,bl,ts,att,fails}}` (+ `rh_home_aso`).

## 10. Открытое (валидировать)
- CAP и веса — первое приближение; подгонять по факту.
- bl на быстрых каналах может быть null (проба завершается позже закачки).
- Существующие инструменты (clash-speedtest и подобные) сортируют по скорость+задержка
  без порога-вышибалы — это подтверждает п.0 (выбор относительный).
