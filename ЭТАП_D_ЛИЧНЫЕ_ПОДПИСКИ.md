# ЭТАП D — ЛИЧНЫЕ ПОДПИСКИ (черновик архитектуры)

> Согласовано в обсуждении (чат Этапа C → задел на D). Уточняется.
> Источник истины по конфигу — `routehub.conf`. Здесь — план личных
> подписок со скоростью. НЕ реализовано (Worker ещё не поднят).

## Цель
- Узел для AI и для основы выбирается по ОТКЛИКУ + СКОРОСТИ закачки.
- Скорость ВИДНА в именах узлов, основа может работать простым fallback.
- Скорость локальна у каждого устройства (вывод 10) → у каждого свой
  отсортированный список → ЛИЧНЫЕ подписки (не одна общая).
- GitHub-токен НИКОГДА на телефоне (вывод 11). На телефоне — только URL.

## Почему именно так (зафиксированные ограничения)
- Имена узлов подписки в Loon неизменяемы → вписать скорость можно
  только на сервере (при сборке списка), не локально.
- Скрипт Loon НЕ может менять URL подписки `[Remote Proxy]` → ключ
  устройства должен быть в самой ссылке, которую Loon дёргает.
- GitHub raw/gist статичны (логику на запрос не выполняют) → нужен
  маленький динамический сервер (Cloudflare Worker / serverless).
- По ОБЩЕЙ ссылке сервер не различит устройства (IP меняется/общий за
  NAT, cookie не хранится, UA одинаковый) → авто-привязка по общей
  ссылке невозможна. Поэтому ключ — в ссылке (`?key=kN`), Loon при
  обновлении дёргает ровно её → привязка держится.

## Компоненты
1. **GitHub** — базовый `routehub.conf` (правится как сейчас, вручную в
   репо). Токен `gist`/repo — в Secrets/в env Worker, НЕ на телефоне.
2. **Worker** (Cloudflare/serverless, бесплатный; альтернативы:
   Pipedream/Apps Script/Firebase+Action). Делает ОБА:
   - `GET /config?key=kN` → берёт базовый routehub.conf из GitHub,
     подставляет персональную ссылку на узлы (для kN) и вписывает kN
     в `[Argument]`/аргумент скрипта (чтобы спидтест помечал заливки),
     отдаёт готовый конфиг.
   - `POST` скорости `{key:kN, nonce:..., speeds:[...]}` → пересобирает
     список узлов для kN (имена со скоростью, сортировка по
     скорости+отклику), кладёт в гист как файл `nodes-kN.txt`
     (один гист, файл на ключ).
3. **Секретный гист** — файлы `nodes-kN.txt` (узлы под каждый ключ) +
   `devices.json` (реестр ключей). Один гист, новые файлы добавляет
   Worker (руками гисты не создавать).

## Реестр устройств (devices.json)
- По ключу: `status` (free/bound/conflict), `label` (чей телефон),
  `nonce` (секрет устройства-владельца), first_seen, last_seen.
- ВСЕГДА ≥1 ключ `free` (запас). Активация free → `bound` + создаётся
  новый `free`.
- `bound` ставить по ПЕРВОЙ заливке скорости (реальное устройство), не
  по скачиванию конфига.
- Диана смотрит devices.json на GitHub → видит, какой ключ свободен,
  даёт его ссылку новому устройству. Выдавать ПО ОДНОМУ.

```json
{
  "k1": {"status":"bound","label":"мой iPhone","nonce":"a1b2","first_seen":"...","last_seen":"..."},
  "k2": {"status":"bound","label":"жена","nonce":"c3d4","first_seen":"...","last_seen":"..."},
  "k3": {"status":"free"}
}
```

## Защита от коллизий (если один ключ выдан дважды)
- Устройство генерит случайный **nonce** (локальный секрет в
  `$persistentStore`), шлёт его в КАЖДОЙ заливке скорости.
- Worker при ПЕРВОЙ заливке для ключа запоминает nonce («владелец»).
  Дальше заливки обязаны совпасть по nonce.
- Заливка с ДРУГИМ nonce на занятый ключ → Worker её ОТКЛОНЯЕТ и ставит
  ключу `status:conflict` в devices.json → Диана видит и выдаёт второму
  устройству свежий ключ. Слипания (смешения скоростей) не происходит.
- ВАЖНО: один и тот же ключ НЕЛЬЗЯ «разделить» на два ID — оба устройства
  шлют один ключ, Worker различит их только по nonce (первый владеет,
  второй отклоняется). Поэтому базовая мера — уникальный ключ на
  устройство + выдавать по одному (реестр free/bound). Nonce — страховка
  от случайной двойной выдачи, не замена.

## Поток
1. Диана даёт устройству ссылку свободного ключа: `…/config?key=k3`
   (вставляется ОДИН раз при установке — как любой конфиг; убрать
   нельзя из-за ограничений Loon).
2. Loon тянет конфиг → Worker отдаёт персональный (узлы = `nodes-k3`).
3. Спидтест на телефоне меряет скорость → читает свой key (из аргумента,
   впечатанного Worker) + свой nonce → POST на Worker.
4. Worker пересобирает `nodes-k3.txt` (сортировка по скорости+отклику),
   помечает k3 `bound` (запоминает nonce), пополняет запас.
5. Loon обновляет конфиг и узлы по keyed-URL → привязка держится.
6. Основа: узлы уже отсортированы Worker → можно fallback (без
   локального селектора). AI: Германия + 2 страны, внутри — по
   скорости+отклику (узлы уже размечены).

## Безопасность
- Токен GitHub только в Worker. На телефоне — keyed-URL + локальный nonce.
- Утечка ключа: чужой увидит узлы ЭТОГО устройства (те же секретные
  узлы, что и так в подписке); слать фейк-скорость не сможет без nonce
  (или вызовет conflict, который ты увидишь). GitHub и другие устройства
  недоступны. Меры: случайные ключи, rate-limit/валидация, ротация.
