## ИСПРАВЛЕНИЯ К ПЛАНУ (ошибки, найденные сверкой)

### И1. `switch-node-after-failure-times` — УСТАРЕЛ
План v2 Раздел 4 (C.1) закладывал этот параметр.
Документация: параметр **устарел** («已弃用»). Loon теперь сам
автоматически определяет доступность узла и переключает по
контексту.
ПРАВКА: убрать строку из [General]. Нативная авто-подстраховка
работает без неё. Health-check её дополняет.

### И2. `ip-mode` — неверное значение
План писал `ip-mode = v4-only`.
Документация: правильно `ipv4-only` (варианты: ipv4-only, dual,
ipv4-preferred, ipv6-preferred).
ПРАВКА: `ip-mode = ipv4-only`.

### И3. `max-timeout` — есть НЕ у всех групп
План (A4) хотел проверить `max-timeout` у url-test.
Документация: `max-timeout` есть у **fallback** и **load-balance**,
у **url-test — НЕТ** (только url, interval, tolerance).
ПРАВКА: убрать вопрос A4 как открытый — ответ известен.

## ПОДТВЕРЖДЕНО ДОКУМЕНТАЦИЕЙ

### П1. Типы групп
- `select` — ручной выбор (наши ИИ-группы, скрипт управляет). ✅
- `url-test` — параметры: url, interval, tolerance(мс). ✅
- `fallback` — параметры: url, interval, max-timeout(мс). ✅
- `load-balance` — url, interval, max-timeout, algorithm. ✅
- Группы можно вкладывать (но мы ограничились 3 уровнями). ✅

### П2. load-balance algorithm — НАХОДКА
Алгоритмы: Random, **PCC**, Round-Robin.
PCC — привязывает запросы одного хоста к одному узлу.
Для обычного трафика PCC может быть лучше url-test: не «дёргает»
активные соединения. Рассмотреть для группы АВТО.

### П3. Типы правил [Rule]
DOMAIN, DOMAIN-SUFFIX, DOMAIN-KEYWORD, IP-CIDR, IP-CIDR6, GEOIP,
IPASN, SRC-PORT, DEST-PORT, PROTOCOL, URL-REGEX, USER-AGENT.

### П4. Производительность правил — НАХОДКА
Документация: DOMAIN, DOMAIN-SUFFIX, IP-CIDR, GEOIP, IPASN,
DEST-PORT, PROTOCOL — ищутся мгновенно при любом количестве.
DOMAIN-KEYWORD, USER-AGENT, URL-REGEX — ЗАМЕДЛЯЮТСЯ с ростом.
ВЫВОД: предпочитать списки на DOMAIN-SUFFIX, избегать больших
списков на DOMAIN-KEYWORD. Влияет на выбор списков (Раздел 16).

### П5. ssid-trigger — подтверждён точно
`ssid-trigger = "loon-wifi5g":DIRECT,"cellular":PROXY,"default":RULE`
Значения режима: DIRECT, PROXY, RULE.
Для нашего разделения сетей (Этап E) — рабочий механизм.

### П6. interface-mode — это «Сетевой интерфейс» со скриншота
Auto / Cellular / Performance / Balance. Глобальный, не для нашей
задачи. Оставляем Auto.

## НАХОДКИ — ДОБАВИТЬ В ПЛАН

### Н1. proxy-test-url (глобальный)
`proxy-test-url = http://cp.cloudflare.com/generate_204`
Глобальный URL теста узлов, если у группы свой не задан.
Документация сама рекомендует cloudflare. Добавить в [General].

### Н2. internet-test-url
`internet-test-url = http://wifi.vivo.com.cn/generate_204`
URL проверки «есть ли вообще интернет». Полезно для детектора
РКН и предварительной проверки канала в спидтесте. Добавить.

### Н3. real-ip — важно для Apple
`real-ip = *.apple.com,*.icloud.com`
Заставляет указанные домены получать настоящий IP (не fakeip).
Помогает iMessage/FaceTime/iCloud работать корректно. Добавить
в [General] — связано с Этапом I (звонки, Apple).

### Н4. hijack-dns
`hijack-dns = *:53,8.8.8.8`
Перехват DNS-запросов приложений, которые лезут мимо Loon.
Может пригодиться для TikTok-модуля (он про DNS). Отметить
в Этапе J как опцию.

### Н5. doh3-server / doq-server
Loon поддерживает DNS over HTTP/3 (`doh3-server = h3://...`) и
DNS over QUIC (`doq-server = quic://...`), не только DoH.
Можно использовать более современный DNS. Опционально.

### Н6. DEST-PORT / PROTOCOL для звонков — НАХОДКА
Правила DEST-PORT и PROTOCOL ищутся мгновенно. Голосовой UDP
можно ловить по портам (как делает voice_ports.list misha-tgshv).
Полезно для Этапа I (звонки).

### Н7. resource-parser
`resource-parser = .../sub-store-parser.loon.min.js`
Парсер подписок Sub-Store. Если формат подписки Lastdep не
распознаётся напрямую — этот парсер помогает. Держать в уме.

### Н8. skip-proxy / bypass-tun
Два механизма обхода: bypass-tun (для TUN), skip-proxy (для HTTP
proxy). Локальные подсети — в оба. Уточнить в [General].

провайдер. От маркера 🙏 зависит вся экономия обходного трафика.

## НАХОДКИ ИЗ Script API + Script Types (проход 2)

### И4. network-changed — ТОЛЬКО ОДИН скрипт (ВАЖНО)
Документация: если несколько скриптов типа network-changed,
вызывается ТОЛЬКО ПЕРВЫЙ в конфиге.
План v2: и ai-region-switch, и спидтест-триггер хотели
реагировать на смену сети. ДВА network-changed НЕ РАБОТАЮТ.
ПРАВКА: один network-changed скрипт-диспетчер, который сам
решает что вызвать (обновить узлы / запустить сотовый замер).
Либо ai-region-switch.js делает обе задачи.

### И5. Таймауты скриптов
Документация: дефолтный таймаут cron/network-changed = 200с.
План местами ставил timeout=30 — мало для сетевых запросов.
ПРАВКА: cron-скрипты timeout=60-120, спидтест можно 200.

### П7. cron — синтаксис подтверждён
"мин час день месяц неделя" или с секундами
"сек мин час день месяц неделя". Параметры: tag, timeout,
argument, enable.

### П8. generic-скрипт — это наш viewer
Запускается вручную из интерфейса, принимает узел/группу как
параметр. ai-switch-viewer.js — generic-тип. Параметр img-url
задаёт иконку.

### Н9. $utils.geoip / ipasn / ipaso — гео ПРЯМО на телефоне
Скрипт может определить страну/ASN IP сам, без сервера.
Основную проверку оставляем серверу, но health-check может
быстро проверить страну узла локально.

### Н10. $config.getConfig() отдаёт ssid и running_model
Скрипт видит: текущий Wi-Fi (ssid), режим Loon (0/1/2),
все группы, текущий выбор каждой группы.
Для разделения сетей (Этап E) — подтверждённый API, надёжнее
догадок. running_model: 0 глоб.директ / 1 правила / 2 глоб.прокси.

### Н11. $config.setRunningModel(model) — переключение режима
Скрипт может переключать режим Loon. Для детектора РКН:
при whitelist можно переключать режим программно.

### Н12. $httpClient alpn:"h2"
Для нескольких запросов к одному хосту рекомендуется alpn h2 —
параллельность. Полезно в спидтесте и health-check.

### И6. Метод переключения группы — ПЕРЕПРОВЕРИТЬ
Документация описывает $config.getConfig(policyName,selectName)
как установку выбора (true/false). Наша диагностика опиралась
на setSelectPolicy. На Этапе A перепроверить, какой метод
реально переключает группу. Добавить в открытые вопросы (A12).

### Н13. $notification.post — пуши с переходом
Поддерживает attach с openUrl (loon://switch), mediaUrl,
clipboard. Для health-check «нужно внимание» — пуш с переходом
в Loon.

## НАХОДКИ ИЗ Node + NodeFilter (проход 3)

### П9. Протоколы Loon (полный список)
ShadowSocks(+shadow-tls,+simpleObfs), ShadowSocksR, VMESS,
VLESS(+reality), Trojan, HTTP/S, Socks5, Wireguard, Hysteria2,
Custom by JS.
ВАЖНО: HTTP/S и Custom by JS НЕ поддерживают UDP. Остальные —
могут (параметр udp=true).

### И7. Hysteria2/Wireguard — сервер их не протестирует
Loon на телефоне поддерживает Hysteria2 и Wireguard. НО наш
серверный тестировщик на Xray их НЕ умеет.
ПРАВКА: если в подписке появятся такие узлы — либо sing-box на
сервере, либо помечать «сервером не протестирован». Учесть в
Разделе 3 (B.4.3).

### П10. udp=true — параметр узла, виден серверу
UDP-поддержка определяется типом протокола + параметром udp.
Сервер может определить (Этап I, звонки) — подтверждено.

### П11. Subscription-Userinfo
Loon читает заголовок подписки: upload/download/total/expire.
Можно показывать пользователю остаток трафика. Опционально.

### П12. Типы фильтров узлов [Remote Filter]
NodeSelect (ручной выбор), NameKeyword (ключевое слово),
NameRegex (регулярка).
Готовые regex-шаблоны (из документации):
  ^.*(A|B)        = A или B
  (A.*B|B.*A)     = есть A и B
  ^(?!.*A)        = НЕ содержит A
  ^(?!.*?B).*A    = есть A, но НЕ содержит B

### Н14. Фильтр для голосовой группы — NodeSelect
Для группы ЗВОНКИ можно сделать NodeSelect — вручную выбрать
узлы, показавшие стабильные звонки на Этапе A. Не regex, а
ручной список проверенных UDP-узлов.

### Н15. Filter-ИИ regex
Для «обычные узлы без обхода»: ^(?!.*?(🙏|Обход)).* —
шаблон подтверждён документацией.

## НАХОДКИ ИЗ DNS + Plugin (проход 4)

### Н16. [Host] / DNS-маппинг — решает TikTok-модуль
Раздел [Host] умеет:
- домен → конкретный DNS-сервер: `domain = server:https://...`
- домен → домен, домен → IP
- DNS по SSID: `ssid:WIFI = server:...`
- домен → ip-mode индивидуально
Для TikTok (Этап J): домены TikTok → свой DNS, остальное → наш
DoH. Конфликта нет, разведены по доменам. Подтверждено.

### Н17. ПЛАГИНЫ LOON = наши «модули» (КРУПНАЯ НАХОДКА)
Плагин Loon — официальный механизм: объединяет rule + rewrite +
script + host + mitm в один файл-подконфиг с переключателем.
Это ровно концепция «модуль», которую мы проектировали.

### Н18. [Argument] — настройки модуля в UI Loon (build 733+)
Плагин может иметь параметры прямо в интерфейсе Loon:
- switch — переключатель вкл/выкл
- select — выбор из вариантов
- input — ввод значения
Параметры читаются скриптом ($argument.arg1) или управляют
cron/enable. То есть интервалы, пороги, размеры батчей —
настраиваются в UI, без правки кода.
ВЫВОД: оформить наши модули (спидтест, TikTok, звонки, детектор
РКН) как плагины с [Argument]. Переключатели и настройки — в UI.
Это закрывает требование «модульность + настройка в интерфейсе».

### И8. Ограничение плагинов — политики правил
Правила ВНУТРИ плагина могут указывать только на: DIRECT,
REJECT-типы, или PROXY (= группа, выбранная юзером при установке).
НЕ могут ссылаться на наши именованные группы (ИИ-Claude и т.д.).
ПРАВКА: основная ИИ-маршрутизация — в ГЛАВНОМ конфиге. В плагины
выносить только самодостаточное (спидтест-скрипт, TikTok DNS,
рекламные правила). Учесть при разбивке на модули.

### П13. REJECT-типы (полный список)
REJECT, REJECT-IMG, REJECT-DICT, REJECT-ARRY, REJECT-DROP.
Для рекламы — REJECT-DROP (подтверждено, тихий отброс).

### Н19. Loon-Gallery — репозиторий плагинов
github.com/Peng-YM/Loon-Gallery — готовые плагины сообщества.
Глянуть на Этапе A: возможно, есть готовое для наших задач.

## ВТОРОЙ ПРОХОД — находки (Policy, Rewrite, Rule подробно)

### П15. REJECT-типы — точное поведение
REJECT (404+пусто), REJECT-IMG (200+1px GIF), REJECT-DICT
(200+пустой JSON), REJECT-ARRY (200+пустой массив), REJECT-DROP
(отброс без ответа — против «шторма повторов»). Реклама —
REJECT-DROP подтверждён.

### П16. Правило может указывать на узел по имени
DOMAIN,google.com,香港01 — прямо на узел. Нам не нужно (узлы
динамические), но зафиксировано.

### П17. Rewrite — только для MITM
Переписывание HTTP/S запросов, работает только с расшифровкой
MITM. RouteHub MITM не использует → Rewrite не нужен.

### Н20. ЛОГИЧЕСКИЕ ПРАВИЛА AND/OR/NOT (3.1.7+) — НАХОДКА
`AND,((подправило),(подправило)),Policy`
`OR,((...),(...)),Policy`
`NOT,((...)),Policy`
Совет docs: если в логике есть домен и IP — IP-подправило
ставить последним (избежать лишних DNS-запросов).
Применение: точное правило для звонков —
AND,((PROTOCOL,UDP),(DEST-PORT,...)),RH-Звонки

### Н21. PROTOCOL — типы (3.1.7+) — ВАЖНО для звонков
PROTOCOL поддерживает: HTTP, HTTPS, TCP, QUIC, STUN, UDP.
КЛЮЧЕВОЕ: STUN — протокол установки связи для звонков
(Telegram/Discord/FaceTime). Правило PROTOCOL,STUN,RH-Звонки —
точно ловит голосовые соединения.
PROTOCOL,QUIC — управление QUIC (связано с вопросом A.7
YouTube): можно PROTOCOL,QUIC,REJECT чтобы заставить TCP.

### Н22. Полная карта раздела Rule
domain_rule, ip_rule, http_rule, port_rule, protocol_rule,
logic_rule, final_rule, sub_rule. Все типы учтены в плане.

### Н23. PROCESS-NAME — НЕ для iOS
PROCESS-NAME (правило по имени процесса) — для desktop-версий,
на iOS неприменимо. Не используем.

Новые находки: Н20-Н23.
